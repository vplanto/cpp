# Лекція 14: Життєвий цикл (RAII), Анатомія методів та Структура проекту

## Мета
Автоматизувати "смерть" об'єктів (RAII), зрозуміти приховану магію ключового слова `this` та навчитися розділяти інтерфейс і реалізацію (`.h` vs `.cpp`).

## Експрес-опитування

1.  **Про гарантії:** Ви виїжджаєте з готелю. Ви маєте самі вимити підлогу, випрати постіль і винести сміття, чи це станеться автоматично після того, як ви здасте ключі на рецепції?
2.  **Про права доступу:** Ви надіслали другу посилання на Google-документ у режимі "Тільки для перегляду" (View-only). Чи зможе він там щось дописати? Чому це важливо для безпеки коду?
3.  **Про імена:** Якби ви були "об'єктом", що б ви використовували, щоб вказати на самого себе в розмові ("Я", "Цей хлопець", "this")?

<details markdown="1">
<summary>Інженерна відповідь</summary>

1.  Це аналог **RAII (SBRM)**. "Здача ключів" (вихід об'єкта з пам'яті) запускає процедуру автоматичного прибирання (Деструктор). Вам не треба пам'ятати про очищення вручну — система гарантує це.
2.  Це аналог **const-методів**. Ви обіцяєте, що метод не змінить об'єкт. Якщо ви спробуєте щось змінити в такому методі, компілятор видасть помилку захисту.
3.  В C++ об'єкт використовує вказівник **`this`**. Це спосіб для методу "дізнатися", з яким саме екземпляром класу він зараз працює.

</details>

## Частина 1: RAII — Ресурс як Об'єкт

Головна фішка C++ — не в тому, як ми створюємо об'єкти, а в тому, як ми **гарантуємо їх знищення**.

### Проблема: "Забудькуватий програміст"
У класичному стилі (як у мові C) ми керували ресурсами вручну. Це завжди веде до витоків (Memory Leaks):
```cpp
void badCode() {
    FILE* f = fopen("data.txt", "r"); 
    if (some_error) return; // ЗАБУЛИ закрити файл! Ресурс "витік".
    fclose(f);
}
```

### Рішення: RAII (SBRM)
Ми прив'язуємо життя ресурсу до життя об'єкта на стеку. Ресурс живе рівно стільки, скільки живе блок `{}` (scope).

```cpp
class FileHandler {
    FILE* f;
public:
    FileHandler(const char* name) { f = fopen(name, "r"); }
    
    // Деструктор — приховані "санітари" пам'яті
    ~FileHandler() { 
        if (f) fclose(f); 
        std::cout << "Closed automatically!\n";
    }
};

void goodCode() {
    FileHandler fh("secret.txt"); // Відкрився в конструкторі
    if (error) return; // fh знищується тут, файл закриється САМ.
}
```

> [!TIP]
> **Золоте правило:** Ніколи не пишіть `delete` або `close()` вручну в логіці програми. Загорніть це в клас, і нехай за вас працює Деструктор.

## Частина 2: Анатомія методу (`this`)

Об'єкт у пам'яті — це **тільки його дані** (`hp`, `x`, `y`). Самі методи (код) існують в одному екземплярі для всіх об'єктів. Щоб метод знав, чиє поле `x` він зараз змінює, компілятор передає в нього прихований вказівник **`this`**.

**Як це бачить компілятор:**
```cpp
// Ви пишете:
p1.setX(10);

// Компілятор робить:
Point_setX(&p1, 10); // Передає адресу об'єкта першим аргументом
```

Вам `this` потрібен лише у двох випадках:
1. Якщо ім'я аргументу збігається з іменем поля: `this->x = x;`
2. Якщо ви хочете повернути посилання на самого себе: `return *this;`

## Частина 3: Const Correctness (Захист даних)

Метод, який тільки читає дані (наприклад, `getX()`), **зобов'язаний** бути помічений словом `const`. Це "запобіжник": у такому методі неможливо змінити жодне поле класу.

```cpp
class Date {
    int day;
public:
    int getDay() const { return day; } // Безпечний (Read-only)
    void setDay(int d) { day = d; }    // Небезпечний (Mutating)
};

void print(const Date& d) {
    std::cout << d.getDay(); // ОК
    // d.setDay(10);        // ПОМИЛКА: об'єкт тільки для читання!
}
```

## Частина 4: Структура проекту (.h vs .cpp)

У великих проектах ми розділяємо **ЩО** робить клас (Інтерфейс) та **ЯК** він це робить (Реалізація).

1.  **`.h` (Header)**: Тільки оголошення (контракт). Його читають люди та компілятор.
2.  **`.cpp` (Implementation)**: Детальний код. Його бачить тільки лінкер.

> [!IMPORTANT]
> **Header Guards:** Завжди пишіть `#pragma once` на початку кожного `.h` файлу. Це захищає від помилок "redefinition", якщо ви підключите один файл кілька разів.

> **💡 Кейс Лінуса Торвальдса та Ядро Linux**
> Хоча `#pragma once` коротший, у розробці Ядра Linux використовують старий стиль `#ifndef`, щоб код можна було скомпілювати будь-яким, навіть 30-річним компілятором. Для сучасних проектів ми обираємо зручність: `#pragma once`.

## Частина 5: Coupling & Cohesion (Архітектура)

Якість вашого коду вимірюється двома метриками:

### 1. Coupling (Зв'язність між модулями) — Прагнемо до LOW
Уявіть, що ваша люстра вдома припаяна намертво до проводів у стіні. Щоб її замінити, треба руйнувати стіну. Це **High Coupling** (погано).
Якщо люстра вмикається в **Розетку** — це стандартний інтерфейс. Ви можете замінити люстру на торшер за секунду. Це **Low Coupling** (ідеал).

### 2. Cohesion (Згуртованість всередині модуля) — Прагнемо до HIGH
Уявіть **Швейцарський ніж**, у якому є лезо, ножиці, ліхтарик, флешка і відкривачка для вина. Якщо зламається ліхтарик — ви відправляєте в ремонт увесь ніж. Це **Low Cohesion** (погано).
**Скальпель** робить тільки одну річ, але ідеально. Це **High Cohesion** (ідеал).

<summary>Відповіді</summary>
