## Роль: High-Frequency Trading Engineer

Ваш код обробляє фінансові транзакції. Поточна версія працює 15 секунд. Конкуренти роблять це за 0.5 секунди.
Вас звільнять, якщо ви не прискорите код мінімум у **10 разів**.

Сьогодні ми не пишемо нову логіку. Ми "лікуємо" стару.

## Експрес-опитування
1.  Що робить команда `std::vector::reserve(1000)`? Чи змінює вона `size()`?
2.  Чому `std::endl` повільніший за `\n`?
3.  Що таке прапорець компілятора `-O2` або `-O3`?

<details markdown="1">
<summary>Інженерна довідка</summary>

1.  Виділяє "сиру" пам'ять наперед, щоб уникнути дорогих переаллокацій. `size()` залишається 0.
2.  `std::endl` робить примусовий `flush` (скидання буфера на диск). Це вбиває продуктивність IO.
3.  Це рівні оптимізації. `-O0` (Debug) — без оптимізацій. `-O3` (Release) — агресивна оптимізація (авто-векторизація, інлайн функцій).

</details>

## Крок 1: The Baseline (Повільний код)

Замість написання коду заміру вручну, створіть окремий файл `timer.hpp` і підключіть його:

**timer.hpp**
```cpp
#pragma once
#include <chrono>
#include <iostream>

struct Timer {
    std::chrono::high_resolution_clock::time_point start;
    Timer() : start(std::chrono::high_resolution_clock::now()) {}
    ~Timer() {
        auto stop = std::chrono::high_resolution_clock::now();
        auto duration = std::chrono::duration_cast<std::chrono::milliseconds>(stop - start);
        std::cout << "Time: " << duration.count() << " ms\n";
    }
};
```

**Вимоги до "поганого" коду:**

1. Читайте файл порядково через `getline`.
2. Додавайте у `vector<string>` через `push_back`.
3. **НЕ використовуйте `reserve`.**
4. Після читання виведіть розмір вектора в консоль, використовуючи `std::endl`.

**main.cpp**
```cpp
#include <iostream>
#include <fstream>
#include <vector>
#include <string>
#include "timer.hpp"

int main() {
    {
        Timer t; 
        
        std::ifstream file("logs.txt");
        std::vector<std::string> lines;
        std::string line;

        while (std::getline(file, line)) {
            lines.push_back(line);
        }

        std::cout << "Lines read: " << lines.size() << std::endl;
    } // Тут спрацює деструктор Timer і виведе час
    return 0;
}
```

> **💡 Чому `.hpp`?**
> Це стандартне розширення для заголовочних файлів **C++** (Header for C++). Воно допомагає відрізнити їх від звичайних `.h`, які використовуються у мові C. Це хороша практика для великих проектів.



```bash
g++ main.cpp -o app
./app
```
*Запустіть і запишіть час (T1).*

**Очікуваний вивід (приклад):**
```text
Lines read: 2000000
Time: 390 ms
```

## Крок 2: Memory Optimization (Reserve)

Вектор за замовчуванням росте як: 1, 2, 4, 8, 16...
Кожне розширення — це виділення нової пам'яті + копіювання всього масиву.

**Завдання:**

1. Перед циклом читання викличте `lines.reserve(2000000)`.

```cpp
#include <iostream>
#include <fstream>
#include <vector>
#include <string>
#include "timer.hpp"

int main() {
    {
        Timer t; 
        std::ifstream file("logs.txt");
        
        std::vector<std::string> lines;
        // Крок 2: Резервуємо пам'ять, щоб уникнути переаллокацій
        lines.reserve(2000000); 

        std::string line;
        while (std::getline(file, line)) {
            lines.push_back(line);
        }

        std::cout << "Lines read: " << lines.size() << std::endl;
    }
    return 0;
}
```

**Приклад виводу (після reserve):**
```text
Lines read: 2000000
Time: 311 ms
```

*Запишіть час (T2). Обчисліть приріст (Speedup).*

## Крок 4: Compiler Optimization (The Free Lunch)

Досі ми змінювали код. Тепер змінимо налаштування збірки.
Якщо ви використовуєте IDE (VS Studio, CLion) — перемкніть конфігурацію з **Debug** на **Release**.
Якщо ви в терміналі (`g++`): додайте прапорець `-O3`.

```bash
g++ main.cpp -o app -O3
./app
```

**Очікуваний вивід (після -O3):**
```text
Lines read: 2000000
Time: 240 ms
```

> **Чому така велика різниця?**
> Прапорець **-O3** вмикає агресивну оптимізацію:
> *   **Inlining:** Компілятор "вклеює" код дрібних функцій прямо в місце виклику, прибираючи накладні витрати на переходи.
> *   **Vectorization (SIMD):** Процесор починає обробляти дані пачками (наприклад, по 4 числа за один такт), використовуючи спеціальні інструкції.
> *   **Loop Unrolling:** Розгортання циклів для зменшення кількості перевірок умов.
> *   **Dead Code Elimination:** Видалення всього коду, який не впливає на результат.
>
> У режимі **Debug** (без прапорців) компілятор фокусується на зручності налагодження, а не на швидкості, тому код працює "в лоб" і повільно.


*Запишіть час (T4). Це має бути найшвидший результат.*

## Завдання із зірочкою: Parsing optimization

У нас є рядки типу `"User123 action=4 time=999"`.
Ваша задача — порахувати суму всіх чисел після `time=`.

1. **Naive:** `stringstream` або `string::find` + `substr` + `stoi`. (Повільно).
2. **Pro:** Пройтися по рядку вручну, знайти останній пробіл, і використати `from_chars` (C++17) або `atoi` для хвоста рядка.

> **💡 Modern C++ Feature: std::from_chars (C++17)**
>
> До C++17 для парсингу чисел з рядків використовували:
> - `atoi()`, `atof()` — C-style, нема перевірки помилок
> - `std::stoi()`, `std::stof()` — C++ style, але **повільні** (locale, exceptions)
> - `stringstream` — дуже повільно
>
> **std::from_chars** — швидкий, без виключень парсинг:
> ```cpp
> #include <charconv>
> 
> const char* str = "12345";
> int value;
> auto result = std::from_chars(str, str + 5, value);
> 
> if (result.ec == std::errc{}) {
>     // Успіх! value == 12345
> }
> ```
>
> **Переваги:**
> - **10-100x швидше** за `std::stoi()`
> - Не виділяє пам'ять
> - Не кидає виключення
> - Не залежить від locale
>
> **Приклад з нашим завданням:**
> ```cpp
> // Знайти "time=" у рядку
> size_t pos = line.find("time=");
> const char* start = line.data() + pos + 5;  // Після "time="
> const char* end = line.data() + line.size();
> 
> int time_value;
> std::from_chars(start, end, time_value);
> ```
>
> **Компіляція:**
> ```bash
> g++ -std=c++17 ...
> ```

Спробуйте реалізувати варіант Pro і заміряти різницю.

## Крок 5: The Golden Rule (Vector vs List)

На лекції ми вивчили правило: **Структура даних = Алгоритм**. Доведемо це експериментально, порівнявши швидкість обходу суцільної пам'яті (`vector`) та розкиданих вузлів (`list`).

**Завдання:**
1. Створіть `std::vector<int>` та `std::list<int>`.
2. Заповніть обидва 10,000,000 випадкових чисел.
3. Заміряйте час сумування всіх елементів для обох структур.

```cpp
#include <iostream>
#include <vector>
#include <list>
#include "timer.hpp"

int main() {
    const int N = 10'000'000;
    
    // Створюємо та заповнюємо структури
    std::vector<int> vec(N, 1);
    std::list<int> lst(vec.begin(), vec.end());

    std::cout << "Benchmarking " << N << " elements...\n";

    {
        std::cout << "[Vector] ";
        Timer t;
        long long sum = 0;
        for (int x : vec) sum += x;
        std::cout << "(done) ";
    }

    {
        std::cout << "[List]   ";
        Timer t;
        long long sum2 = 0;
        for (int x : lst) sum2 += x;
        std::cout << "(done) ";
    }

    return 0;
}
```

Компілюйте обов'язково з прапорцем `-O3`, щоб побачити справжню різницю:
```bash
g++ main.cpp -o bench -O3
./bench
```

**Очікуваний результат (приклад з -O3):**
```text
Benchmarking 10000000 elements...
[Vector] Time: 3 ms
(done), sum =  10000000
[List]   Time: 31 ms
(done), sum2 = 10000000
```

**Несподівано отримали 0 мс? Чому так?**
Це ще одна "магія" оптимізації, яка називається **Dead Code Elimination** (видалення мертвого коду). Компiлятор бачить, що ви рахуєте суму, але **ніде її не використовуєте**. Отже, з точки зору компілятора, цей цикл не робить нічого корисного, і його можна просто видалити.

**Як побачити реальні цифри?**
Виведіть результати сум (`sum` та `sum2`) в самому кінці `main`. Тільки якщо результат роботи алгоритму кудись виводиться або десь використовується, компілятор буде змушений чесно виконати цикл. 

### ⚠️ Пастка: "Danger Zone"

При спробі вивести результат легко припуститися двох критичних помилок:

**1. Pointer Arithmetic (Segmentation Fault)**
Якщо ви випадково напишете:
`std::cout << "sum = " + sum;` // ПОМИЛКА!
Програма впаде з помилкою `Segmentation fault`. 
**Чому?** Бо `"sum = "` — це вказівник на початок рядка. Оператор `+` додає до цього вказівника ваше число (наприклад, 10,000,000). Програма намагається прочитати рядок за адресою, яка знаходиться далеко за межами дозволеної пам'яті.
**Рішення:** Завжди використовуйте оператор `<<`.

**2. Замір виводу замість алгоритму**
Якщо ви виведете суму **всередині** блоку, де працює `Timer`, ваші результати будуть майже однаковими (наприклад, 105 ms та 119 ms).
**Чому?** Вивід у консоль — це надзвичайно повільна операція. Вона займає 99% часу, і на її фоні різниця у швидкості пам'яті стає непомітною.
**Рішення:** Спочатку закривайте блок з `Timer` (щоб він вивів час), а вже потім виводьте суму.

*Після того, як ви додасте вивід суми **ПІСЛЯ** заміру, ви побачите реальну різницю: вектор "пролітає" за лічені мілісекунди (3-5 мс), тоді як список працює в 10-20 разів повільніше (30-50 мс).*

### 🧠 Як не помилитися при замірах (Checklist)

Ви помітили, як часто ми "спотикалися" під час практикуму? Не той scope (заміряли вивід), не той оператор (Pointer Arithmetic), або компілятор взагалі видаляв наш код. Ось список правил, як робити бенчмаркінг свідомо:

1.  **Використовуйте результат:** Завжди виводьте фінальне значення (наприклад, суму) **після** заміру. Якщо результат не використовується, компілятор просто видалить ваш код (**Dead Code Elimination**).
2.  **Розділяйте логіку та I/O:** Ніколи не ставте `std::cout` всередині блоку `Timer`. Вивід у консоль настільки повільний, що він "з'їсть" всю різницю між алгоритмами.
3.  **Release-режим — це база:** Завжди порівнюйте швидкість з прапорцем **`-O3`**. Без нього "шум" від неоптимізованого C++ (ітератори, виклики функцій) приховає реальну продуктивність.
4.  **Слідкуйте за типами:** Пам'ятайте про пастку `+` для рядкових літералів. Використовуйте `<<` для виводу чисел.
5.  **Чистота експерименту:** Під час замірів закрийте важкі програми (браузер, ігри), щоб вони не забирали ресурси CPU.

## Контрольні питання

1. `vector::reserve(N)` і `vector::resize(N)` — в чому різниця? Що змінюють `size()` і `capacity()`?
2. Чому `std::endl` повільніший за `"\n"`? У яких ситуаціях `flush` все ж потрібен?
3. Чому обхід `std::vector` зазвичай значно швидший за обхід `std::list` тієї ж довжини? Як це пов'язано з поняттям Cache Line?
4. Прапорець `-O3` прискорив код у 6 разів без змін у логіці. Чому ж ми не компілюємо завжди з `-O3`?

<details markdown="1">
<summary>Відповіді для самоперевірки</summary>

1. `reserve(N)` виділяє пам'ять (збільшує `capacity`), але **не змінює `size()`** і не додає елементів. `resize(N)` змінює і `size()`, і `capacity`, додаючи або видаляючи елементи. Після `reserve` вектор «порожній», але без перерозподілу при `push_back` до N елементів.
2. `std::endl` виводить `'\n'` **і** примусово скидає буфер IO на диск (`flush`). Це дорога системна операція. `"\n"` лише записує символ у буфер. `flush` потрібен, коли важливо, щоб дані точно потрапили на диск до збою (логи, критичні повідомлення).
3. Вектор зберігає дані суцільним блоком у пам'яті, що дозволяє процесору завантажувати їх у кеш пачками (**Cache Lines**) заздалегідь. У списку елементи розкидані по всій RAM, тому кожен перехід до наступного вузла викликає **Cache Miss** — процесор змушений чекати сотні тактів, поки дані прийдуть із повільної пам'яті.
4. `-O3` ускладнює налагодження: компілятор переставляє інструкції, інлайнить функції, видаляє «зайві» змінні — стек-трейс стає нечитабельним, брейкпоінти — ненадійними. У Debug-режимі (`-O0`) код повільніший, але відповідає вихідному тексту.

</details>
