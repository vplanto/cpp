## 🧩 Частина 1: The Circular Train Problem (Logic & Complexity)

**Контекст:** Ви замкнені у потязі, який з'єднаний у кільце (останній вагон з'єднаний з першим). Ви не знаєте, скільки вагонів ($N$). У кожному вагоні є лампочка, яку можна ввімкнути (ON) або вимкнути (OFF).
Початковий стан лампочок — випадковий.

**Задача:** Порахувати кількість вагонів $N$.
**Обмеження:**
1.  Ви можете вмикати/вимикати світло ТІЛЬКИ у поточному вагоні.
2.  Ви можете переходити вперед або назад.
3.  Ви не бачите світло в інших вагонах, поки не зайдете в них.

### Алгоритм дій (Рішення: [CabinCount.cpp](https://github.com/vplanto/cpp/blob/main/xx-taste/CabinCount.cpp))

Ми не можемо просто йти вперед і рахувати, бо не знаємо, коли повернулися на старт (вагони однакові).
Нам потрібен **Маркер**.

**Алгоритм:**
1.  Вмикаємо світло в поточному вагоні (Старт).
2.  Йдемо вперед на $K$ кроків (спочатку $K=1$).
3.  Вимикаємо світло в цьому вагоні ($K$-тому).
4.  Повертаємося назад на $K$ кроків (до Старту).
5.  Перевіряємо: **Світло горить?**
    * **ТАК:** Значить, той вагон, де ми вимкнули світло, — це НЕ наш стартовий вагон. Ми ще не пройшли коло. Збільшуємо $K$ і повторюємо.
    * **НІ:** Значить, коли ми вимкнули світло "там", воно згасло "тут". Це один і той самий вагон! Ми пройшли коло. **Відповідь = K.**

### Завдання
Реалізуйте цей алгоритм.
Вам надано симулятор потяга (масив `vector<bool> train`).

```cpp
// Функції, які вам доступні (API):
void setLight(bool on);   // Вкл/Викл
bool getLight();          // Перевірити стан
void moveForward();       // Крок вперед
void moveBackward();      // Крок назад

```

**Питання:**
Яка алгоритмічна складність цього рішення?

* Ми робимо 1 крок вперед, 1 назад.
* Потім 2 вперед, 2 назад.
* ...
* Потім $N$ вперед, $N$ назад.
* Сума кроків: $2 \cdot (1 + 2 + \dots + N) = N(N+1)$.
* **Результат:** $O(N^2)$. Це квадратичний алгоритм. Чи можна швидше? (Подумайте вдома).

## 🎨 Частина 2: Linus Torvalds та "Good Taste" (Linked List)

Лінус Торвальдс (творець Linux) якось сказав:

> *"I will judge you by how you remove an element from a linked list."*

Він навів приклад двох підходів до видалення елемента зі зв'язного списку.

### Варіант 1: "Student Style" (Bad Taste)

Більшість людей пишуть так. Ми маємо окремо обробити випадок, коли видаляємо **перший** елемент (Head), бо у нього немає "попереднього" (`prev`).

```cpp
void remove_bad(Node* entry) {
    Node* prev = nullptr;
    Node* walk = head;

    // 1. Шукаємо елемент
    while (walk != entry) {
        prev = walk;
        walk = walk->next;
    }

    // 2. Видаляємо
    if (!prev) {
        // Окремий if для голови!
        head = entry->next;
    } else {
        // Стандартний випадок
        prev->next = entry->next;
    }
}

```

**Проблема:** Зайвий `if`. Ускладнення логіки (Edge Case).

### Варіант 2: "Linus Style" (Good Taste)

Використовуємо **вказівник на вказівник** (`Node**`).
Ми не зберігаємо вказівник на *ноду* (`prev`), ми зберігаємо адресу *місця, звідки посилаються* на поточну ноду.

* Для першої ноди — це адреса змінної `head`.
* Для інших — це адреса поля `next` попередньої ноди.

Для комп'ютера це одне й те саме місце в пам'яті!

```cpp
void remove_tasteful(Node* entry) {
    // indirect - це вказівник на вказівник, який зараз "тримає" нашу ноду
    Node** indirect = &head; 

    // Шукаємо, поки (*indirect) не стане нашим entry
    while ((*indirect) != entry) {
        indirect = &(*indirect)->next; // Беремо адресу поля next
    }

    // Видаляємо. Жодних if!
    // Ми просто переписуємо "вхідний лінк" на наступний елемент.
    *indirect = entry->next;
}

```

### Завдання

1. Намалюйте на папері, як працює `Node** indirect`, коли ми видаляємо перший елемент.
2. Реалізуйте обидва варіанти у `main.cpp` і переконайтесь, що вони працюють.

## Частина 3: Container Wars (Cache Locality)

Доведемо те, що вивчили в Лекції 19.

**Завдання:**

1. Створіть `std::vector<int>` та `std::list<int>`.
2. Заповніть обидва 100 000 випадкових чисел.
3. **Виміряйте час сумування** всіх елементів (прохід від початку до кінця).

```cpp
// Підказка для заміру (використовуйте Timer.hpp)
{
    Timer t; 
    long long sum = 0;
    for (int x : my_container) sum += x;
}

```

**Очікуваний результат:** `vector` має бути в 2-10 разів швидшим за `list` (навіть без `-O3`).
**Причина:** Cache Misses. `list` змушує процесор стрибати по пам'яті, `vector` — ні.

<details markdown="1">
<summary>Приклад рішення (Benchmark)</summary>

```cpp
#include <iostream>
#include <vector>
#include <list>
#include "timer.hpp"

int main() {
    const int N = 100'000;
    
    // 1. Створюємо та заповнюємо структури
    std::vector<int> vec;
    vec.reserve(N);
    for(int i=0; i<N; ++i) vec.push_back(rand() % 100);

    std::list<int> lst(vec.begin(), vec.end());

    // 2. Вимірюємо час сумування
    {
        std::cout << "[Vector] ";
        Timer t;
        long long sum = 0;
        for (int x : vec) sum += x;
        // Використовуємо суму, щоб оптимізатор не видалив цикл
        std::cout << "sum=" << sum << " ";
    }

    {
        std::cout << "[List]   ";
        Timer t;
        long long sum = 0;
        for (int x : lst) sum += x;
        std::cout << "sum=" << sum << " ";
    }

    return 0;
}
```
</details>

## Definition of Done

1. **Потяг:** Алгоритм знаходить правильне число вагонів для `N=5` і `N=15`.
2. **Складність:** Ви написали, що алгоритм потяга — $O(N^2)$ (сума арифметичної прогресії).
3. **Список:** Ви реалізували функцію `remove_elegant` з використанням подвійного вказівника (`Node**`).
4. **Бенчмарк:** Ви отримали лог, де `Vector` перемагає `List` у швидкості читання.

## Контрольні питання

1. Алгоритм пошуку у потязі робить 1+1, 2+2, …, N+N кроків. Яка його складність у нотації Big O? Чи існує алгоритм з кращою складністю для цієї задачі?
2. У чому принципова різниця між `prev` (вказівник на ноду) і `indirect` (вказівник на вказівник) у двох версіях `remove`? Чому `indirect` позбавляє нас від окремої перевірки Head?
3. Порівняйте `std::vector` і `std::list` за трьома критеріями: вставка в середину, доступ за індексом, cache locality. Для яких задач кожен з них підходить краще?
4. Що таке «Dangling Pointer»? Наведіть приклад, як він може виникнути при некоректному копіюванні об'єкта з динамічною пам'яттю.

<details markdown="1">
<summary>Відповіді для самоперевірки</summary>

1. Сума кроків: 2·(1+2+…+N) = N·(N+1) → **O(N²)**. Для цієї конкретної задачі з обмеженнями (ми можемо лише рухатись і перемикати світло) швидше навряд чи можливо, але якби ми могли «мітити» вагони якось інакше — можна знайти O(N).
2. `prev` вказує на **вузол** перед тим, що видаляємо. Для Head'а попереднього вузла немає, тому потрібен `if`. `indirect` вказує на **змінну**, що містить адресу нашого вузла: для Head це сама змінна `head`, для інших — поле `next` попередника. В обох випадках `*indirect = entry->next` робить те саме — переписує посилання. Окремий `if` не потрібен.
3. `vector`: вставка в середину O(N) (зсув елементів), доступ за індексом O(1), відмінна cache locality (суцільний блок пам'яті). `list`: вставка в середину O(1) (перелинковування), доступ за індексом O(N), погана cache locality (вузли розкидані по heap). `vector` — для читання та пошуку; `list` — для частих вставок/видалень у середину.
4. Dangling Pointer — вказівник, що вказує на вже звільнену пам'ять. Виникає при shallow copy: обидва об'єкти `s1` і `s2` мають поле `grades`, яке вказує на **той самий** масив. Після `delete[] s1.grades` вказівник `s2.grades` стає «висячим» — звернення до нього є UB (undefined behavior).

</details>
