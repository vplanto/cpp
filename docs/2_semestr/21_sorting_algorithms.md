# Лекція 21: Алгоритми сортування — QuickSort, MergeSort та IntroSort

[← Лекція 20](20_lists_vector.md) | [Index](index.md) | [Далі: Лекція 22 →](22_graphs_search.md)

## Мета

Зрозуміти, як працюють класичні алгоритми сортування "під капотом". Навчитися розрізняти стабільні та нестабільні алгоритми. Дізнатися, чому `std::sort` працює не так, як ви думаєте.

## Експрес-опитування

1. **Ситуація:** Ви пишете систему "Скасування дій" (Undo). Яку комбінацію АТД та реалізації ви оберете: `Stack` на базі `vector` чи `Stack` на базі `list`?
2. **Ситуація:** Ви розробляєте систему для Швидкої допомоги. Вам треба завжди бачити пацієнта з найвищим пріоритетом (найважчий стан) за $O(1)$. Який АТД та яку структуру даних ви оберете?
3. **Ситуація:** У RAM уже є `std::vector` з мільярдом записів. Їх треба впорядкувати, але **додаткової пам'яті майже немає** — другий масив такого ж розміру виділити не вийде. Що оберете: **скопіювати всі дані в другий повнорозмірний буфер** і там збирати результат, чи **переставляти елементи лише в наявному `vector`** (без другого масиву на всі $N$ записів)?

<details markdown="1">
<summary>Інженерна відповідь</summary>

1. **Stack на базі vector (з-поміж цих двох).** Він забезпечує кращу кеш-локальність, ніж `list`. Однак **у реаліях C++** дефолтним контейнером для `std::stack` є `std::deque`. Чому? `std::vector` при переповненні capacity змушений робити реалокацію та дороге копіювання/переміщення всіх існуючих елементів. Натомість `std::deque` виділяє пам'ять чанками (сторінками). Тому додавання елементів ніколи не призводить до копіювання старих даних, і при цьому зберігається цілком прийнятна кеш-локальність (дані всередині чанка лежать суцільно).
2. **Priority Queue (АТД)** реалізований через **Binary Heap (Купу) на базі масиву**. Це забезпечує доступ до вершини за $O(1)$ та чудову кеш-локальність (дані лежать суцільно).
3. **Переставляти в наявному `vector`.** При гігантських $N$ дублювати весь масив ($O(N)$ додаткової RAM) часто неможливо. На лекції побачимо алгоритми **in-place** (на місці) — зокрема QuickSort; вони не вимагають окремої копії на весь об'єм даних (на відміну від MergeSort).

</details>

## Частина 1: Базис для порівняння (O(N^2) алгоритми)

Перед тим, як вивчати швидкі алгоритми, подивимося, чому повільні — **повільні**.

### Bubble Sort (Сортування бульбашкою)

Ідея: Проходимо масив N разів. Кожного разу "спли вливаєм" найбільший елемент вправо.

<div markdown="0" style="margin:1rem 0;">
<iframe src="assets/bubble_sort.html" title="Bubble Sort — покрокова візуалізація" width="100%" height="680" style="display:block;width:100%;min-height:560px;height:68vh;max-height:820px;border:1px solid #d0d7de;border-radius:8px;background:#0f1419;" loading="lazy"></iframe>
<p style="margin-top:0.5rem;font-size:0.9rem;color:#57606a;"><em>Покрокова візуалізація:</em> підсвічування рядків коду, порівняння та обміни в масиві — «Далі» / «Назад» або <kbd>←</kbd> <kbd>→</kbd>. <a href="assets/bubble_sort.html" target="_blank" rel="noopener">Відкрити в окремій вкладці</a> · <a href="https://github.com/vplanto/cpp/tree/main/docs/2_semestr/assets">вихідні файли</a></p>
</div>

```cpp
void bubbleSort(std::vector<int>& arr) {
    int n = arr.size();
    
    // N проходів
    for (int i = 0; i < n - 1; i++) {
        // Кожного разу зменшуємо область пошуку
        for (int j = 0; j < n - i - 1; j++) {
            if (arr[j] > arr[j + 1]) {
                std::swap(arr[j], arr[j + 1]); // "Бульбашка" спливає
            }
        }
    }
}
// Складність: O(N^2)
// Стабільність: Так
```

**Стабільність (stability):** алгоритм **стабільний**, якщо елементи з **однаковим ключем** після сортування лишаються у **тому самому відносному порядку**, що й у вхідному масиві. У Bubble Sort умова `arr[j] > arr[j + 1]` — **строга** нерівність: сусіди з однаковим значенням **не** міняються місцями, тому «раніший» елемент залишається лівіше «пізнішого». Це має значення, коли ключ — лише частина даних: наприклад, сортуєте транзакції за сумою, а при однаковій сумі потрібно зберегти хронологію внесення в базу.

**Чому повільно?** Навіть якщо масив майже відсортований, алгоритм все одно робить N^2 порівнянь.

### Selection Sort (Сортування вибором)

Ідея: Шукаємо мінімальний елемент і ставимо його на початок. Повторюємо для хвоста масиву.

<div markdown="0" style="margin:1rem 0;">
<iframe src="assets/selection_sort.html" title="Selection Sort — покрокова візуалізація" width="100%" height="680" style="display:block;width:100%;min-height:560px;height:68vh;max-height:820px;border:1px solid #d0d7de;border-radius:8px;background:#0f1419;" loading="lazy"></iframe>
<p style="margin-top:0.5rem;font-size:0.9rem;color:#57606a;"><em>Покрокова візуалізація:</em> пошук <code>minIdx</code> у хвості, порівняння та обмін на позицію <code>i</code> — «Далі» / «Назад» або <kbd>←</kbd> <kbd>→</kbd>. <a href="assets/selection_sort.html" target="_blank" rel="noopener">Відкрити в окремій вкладці</a> · <a href="https://github.com/vplanto/cpp/tree/main/docs/2_semestr/assets">вихідні файли</a></p>
</div>

```cpp
void selectionSort(std::vector<int>& arr) {
    int n = arr.size();
    
    for (int i = 0; i < n - 1; i++) {
        int minIdx = i;
        
        // Шукаємо мінімум в невідсортованій частині
        for (int j = i + 1; j < n; j++) {
            if (arr[j] < arr[minIdx]) {
                minIdx = j;
            }
        }
        
        std::swap(arr[i], arr[minIdx]);
    }
}
// Складність: O(N^2)
// Стабільність: Ні (swap може порушити порядок)
```

### Insertion Sort (Сортування вставками)

Ідея: Умовно ділить масив на дві частини — вже відсортовану ліву та невідсортовану праву. На кожному кроці береться перший елемент з невідсортованої частини і вставляється на правильне місце у лівій відсортованій частині (процес дуже схожий на те, як гравець впорядковує отримані карти у своїй руці).

```cpp
void insertionSort(std::vector<int>& arr) {
    int n = arr.size();
    for (int i = 1; i < n; i++) {
        int key = arr[i];
        int j = i - 1;
        
        // Зсуваємо елементи, які більші за key, на одну позицію вправо
        while (j >= 0 && arr[j] > key) {
            arr[j + 1] = arr[j];
            j = j - 1;
        }
        arr[j + 1] = key;
    }
}
// Складність: O(N^2) у гіршому/середньому випадку, O(N) на майже відсортованих даних
// Стабільність: Так
```

## Частина 2: QuickSort — Divide and Conquer

Алгоритм, вигаданий Tony Hoare у 1959 році. Досі один з найшвидших на практиці.

</div>

### Ідея

1. Вибрати **pivot** (опорний елемент, наприклад, середній).
2. **Partition:** Переставити елементи так, щоб ліворуч від pivot були менші, праворуч — більші.
3. **Рекурсивно** відсортувати ліву та праву частини.

```cpp
int partition(std::vector<int>& arr, int low, int high) {
    int pivot = arr[high]; // Беремо останній елемент як pivot (проста стратегія)
    int i = low - 1;
    
    for (int j = low; j < high; j++) {
        if (arr[j] <= pivot) {
            i++;
            std::swap(arr[i], arr[j]);
        }
    }
    
    std::swap(arr[i + 1], arr[high]);
    return i + 1; // Позиція pivot
}

void quickSort(std::vector<int>& arr, int low, int high) {
    if (low < high) {
        int pi = partition(arr, low, high);
        
        // Рекурсивно сортуємо ліву та праву частини
        quickSort(arr, low, pi - 1);
        quickSort(arr, pi + 1, high);
    }
}
```

### Складність

- **Середній випадок:** O(N log N) — кожен рівень рекурсії обробляє N елементів, глибина дерева — log N.
- **Найгірший випадок:** O(N^2) — якщо pivot завжди вибирається як мінімум/максимум (відсортований масив).

### Як уникнути O(N^2)?

1. **Median-of-Three:** Вибрати pivot як медіану з `arr[low]`, `arr[mid]`, `arr[high]`.
2. **Randomized QuickSort:** Вибрати випадковий pivot. Математично доведено, що очікуваний час — O(N log N).

**Стабільність:** Ні (partition може переставляти рівні елементи).

## Частина 3: MergeSort — Гарантований O(N log N)

Алгоритм від **John von Neumann** (1945) — математика, який також розробив "архітектуру фон Неймана", що стала базою для побудови майже всіх сучасних комп'ютерів. Ідеальний алгоритм, коли потрібна сувора передбачуваність часу виконання.

### Інженерний погляд: Чому однакові O(N log N) працюють з різною швидкістю?

Математично QuickSort, MergeSort та HeapSort мають однакову асимптотику $O(N \log N)$ (у середньому або гарантованому випадках). Але на практиці їхня швидкість кардинально відрізняється через **архітектуру сучасних процесорів**.

Справа в тому, що читання даних з оперативної пам'яті (RAM) є вкрай повільним. Тому процесор використовує **CPU Cache** (L1, L2, L3) — надшвидку пам'ять малого об'єму. Коли ви читаєте `arr[i]`, процесор завантажує в кеш не один елемент, а цілий блок сусідніх байтів (cache line).

1. **QuickSort (Кеш-френдлі):** Під час `partition` алгоритм читає масив **суворо послідовно** (зліва направо і справа наліво). Дані ідеально лягають у кеш-лінії, процесор легко передбачає розгалуження (branch prediction), а кількість дорогих "промахів кешу" (cache misses) мінімальна. Тому він найшвидший на практиці.
2. **MergeSort:** Також читає масив послідовно під час злиття, що чудово для кешу. Проте постійне копіювання даних між оригінальним і тимчасовим масивом ($O(N)$ пам'яті) створює додаткове навантаження на шину пам'яті.
3. **HeapSort (Вбивця кешу):** Алгоритм будує дерево всередині масиву. Щоб знайти дітей вузла `i`, він стрибає на індекси `2*i` та `2*i + 1`. На великих масивах ці стрибки хаотично розкидані по всій пам'яті. Кожен стрибок — це гарантований **Cache Miss**, через що процесор змушений простоювати в очікуванні даних з RAM. Саме тому на практиці HeapSort працює помітно повільніше за QuickSort.

### Ідея

<div markdown="0" style="margin:1rem 0;">
<iframe src="assets/merge_sort.html" title="Merge Sort — покрокова візуалізація" width="100%" height="680" style="display:block;width:100%;min-height:560px;height:68vh;max-height:820px;border:1px solid #d0d7de;border-radius:8px;background:#0f1419;" loading="lazy"></iframe>
<p style="margin-top:0.5rem;font-size:0.9rem;color:#57606a;"><em>Покрокова візуалізація:</em> divide → conquer → <code>merge</code>, активний підмасив і локально відсортовані сегменти (пунктир) — «Далі» / «Назад» або <kbd>←</kbd> <kbd>→</kbd>. <a href="assets/merge_sort.html" target="_blank" rel="noopener">Відкрити в окремій вкладці</a> · <a href="https://github.com/vplanto/cpp/tree/main/docs/2_semestr/assets">вихідні файли</a></p>
</div>

1. **Divide:** Розділити масив навпіл.
2. **Conquer:** Рекурсивно відсортувати кожну половину.
3. **Merge:** Злити дві відсортовані половини в один масив.

```cpp
void merge(std::vector<int>& arr, int left, int mid, int right) {
    int n1 = mid - left + 1;
    int n2 = right - mid;
    
    // Тимчасові масиви
    std::vector<int> L(n1), R(n2);
    
    for (int i = 0; i < n1; i++) L[i] = arr[left + i];
    for (int i = 0; i < n2; i++) R[i] = arr[mid + 1 + i];
    
    // Злиття
    int i = 0, j = 0, k = left;
    while (i < n1 && j < n2) {
        if (L[i] <= R[j]) {
            arr[k++] = L[i++];
        } else {
            arr[k++] = R[j++];
        }
    }
    
    // Копіюємо залишок
    while (i < n1) arr[k++] = L[i++];
    while (j < n2) arr[k++] = R[j++];
}

void mergeSort(std::vector<int>& arr, int left, int right) {
    if (left < right) {
        int mid = left + (right - left) / 2;
        
        mergeSort(arr, left, mid);
        mergeSort(arr, mid + 1, right);
        merge(arr, left, mid, right);
    }
}
```

### Переваги

- **Завжди O(N log N)** — незалежно від вхідних даних.
- **Стабільне** сортування (при правильній реалізації `merge`).

### Недоліки

- **Додаткова пам'ять:** Потрібен допоміжний масив розміром N (O(N) space).
- **Повільніше за QuickSort** на практиці через більше копіювань даних.

## Частина 4: IntroSort — Гібрид у `std::sort`

**Інженерне рішення:** Взяти QuickSort, але захиститися від O(N^2).

### Компоненти IntroSort

IntroSort (Introspective Sort) — це розумний гібрид трьох різних алгоритмів:
- **QuickSort** — основний рушій (надшвидкий завдяки кеш-френдлі послідовному доступу).
- **Insertion Sort** — для фінального «полірування» мікро-масивів без накладних витрат.
- **HeapSort** — як «план Б» для гарантованого $O(N \log N)$. 

> **Що таке HeapSort (Пірамідальне сортування)?** 
> Він будує **бінарну купу** прямо всередині масиву, де найбільший елемент завжди знаходиться у "корені". Потім $N$ разів алгоритм переміщує цей корінь в кінець масиву (на фінальне місце) та відновлює властивість купи (`heapify`). Складність **завжди** $O(N \log N)$, алгоритм сортує суворо **на місці** (без додаткової пам'яті). Проте, через постійні хаотичні стрибки індексами `2i` та `2i+1`, він постійно вимиває процесорний кеш і на практиці працює повільніше за QuickSort.

### Алгоритм роботи:

1. **Основний цикл:** Починаємо сортувати масив за допомогою звичайного **QuickSort**.
2. **Захист від $O(N^2)$:** Якщо під час QuickSort глибина рекурсії перевищує `2 * log(N)` (це означає, що півоти обираються вкрай невдало і алгоритм деградує), система примусово перемикається на **HeapSort** для цього підмасиву.
3. **Оптимізація мікроданих:** Якщо розмір підмасиву стає меншим за 16 елементів, QuickSort зупиняється. Замість важкої рекурсії викликається **Insertion Sort**, який блискавично досортовує ці короткі шматочки.

```cpp
// Псевдокод std::sort (спрощено)
template<typename T>
void introsort(std::vector<T>& arr, int maxDepth) {
    if (arr.size() < 16) {
        insertionSort(arr); // Простий алгоритм для малих N
        return;
    }
    
    if (maxDepth == 0) {
        heapSort(arr); // Захист від O(N^2)
        return;
    }
    
    int pivot = partition(arr); // QuickSort partition
    introsort(arr_left, maxDepth - 1);
    introsort(arr_right, maxDepth - 1);
}
```

**Складність:** Гарантований O(N log N) у найгіршому випадку.

**Стабільність:** Ні. Для стабільного сортування є `std::stable_sort` (використовує MergeSort).

### Інженерне обмеження: З чим працює `std::sort`?

Лекція багато говорить про `std::sort`, але є критичний C++ факт: **цей алгоритм вимагає Random Access Iterators (ітераторів довільного доступу)**.
Він працює тільки з тими контейнерами, де можна миттєво "стрибнути" на будь-яку позицію (наприклад, `arr[i]`). 

- Ви **можете** відсортувати `std::vector`, `std::array`, `std::deque`.
- Якщо ви спробуєте викликати `std::sort` для `std::list` (або `std::forward_list`), **код просто не скомпілюється**.

Для списків у C++ передбачено вбудований метод `std::list::sort()`, який під капотом реалізує стабільний **Merge Sort**. Він ідеально підходить для зв'язних списків, оскільки не вимагає додаткової пам'яті $O(N)$ (на відміну від MergeSort для масивів), адже злиття відбувається простим перекиданням вказівників між вузлами.

| **Bubble Sort** | O(N^2) | O(N^2) | O(1) | Так | Ніколи (тільки навчання) |
| **Selection Sort** | O(N^2) | O(N^2) | O(1) | Ні | Ніколи |
| **Insertion Sort** | O(N^2)* | O(N^2) | O(1) | Так | Малі $N$ ($<16$), майже відсортовані дані; хвіст IntroSort |
| **QuickSort** | O(N log N) | O(N^2) | O(log N) | Ні | Загальне призначення, великі дані |
| **HeapSort** | O(N log N) | O(N log N) | O(1)** | Ні | Гарантія $O(N \log N)$ in-place; запасний варіант IntroSort |
| **MergeSort** | O(N log N) | O(N log N) | O(N) | Так | Коли потрібна стабільність або гарантований час |
| **IntroSort** | O(N log N) | O(N log N) | O(log N) | Ні | **std::sort** — дефолт для C++ |
| **std::stable_sort** | O(N log N) | O(N log N) | O(N) | Так | Коли важливий порядок рівних елементів |

\* Insertion Sort: **O(N)** на майже відсортованому масиві, але в таблиці зазвичай вказують гірший аналітичний випадок $O(N^2)$.

\** HeapSort: Сортує суворо in-place. Функцію `heapify` можна написати рекурсивно (тоді пам'ять буде $O(\log N)$ на стек викликів), але в інженерних реалізаціях (як-от у стандартній бібліотеці C++) її завжди пишуть ітеративно (через цикл `while`). Саме ітеративний HeapSort гарантує абсолютне $O(1)$ додаткової пам'яті, що робить його ідеальним вибором для систем із жорсткими обмеженнями пам'яті (наприклад, embedded системи).

## Інженерний погляд: Сортування в епоху ШІ

Уявіть ситуацію: ви отримали завдання відсортувати логи фінансових транзакцій сервера за часом. Ви відкриваєте ChatGPT або Claude і пишете: *"Напиши алгоритм сортування на C++"*. ШІ за мілісекунди видає вам ідеально написаний і відформатований код QuickSort або MergeSort. Здається, що робота виконана.

А тепер ускладнимо задачу. Ваш Tech Lead каже: 
> *"У нас гігабайти логів. Сервер має дуже обмежену оперативну пам'ять (RAM), ми не можемо дозволити собі виділити ще стільки ж пам'яті для копіювання масиву. До того ж, критично важливо, щоб транзакції, які відбулися в одну й ту саму мілісекунду, залишилися в тому ж порядку, в якому вони були записані початково"*.

Якщо ви просто використаєте дефолтний згенерований `std::sort` або `QuickSort`, ви **провалите завдання**. 
Чому? 
1. **QuickSort** (і `std::sort`, який базується на IntroSort) сортує *на місці* (in-place) — це вирішує проблему з пам'яттю. Але він **нестабільний**! Ваші транзакції з однаковим часом хаотично перемішаються, і відновити хронологію буде неможливо.
2. Ви повертаєтесь до ШІ: *"Тоді дай стабільне сортування"*. ШІ видає **MergeSort** (або ви берете `std::stable_sort`). Але MergeSort вимагає $O(N)$ додаткової пам'яті для злиття. Вашому серверу не вистачить RAM, і програма впаде з `Out Of Memory`.

**Ось тут починається справжня інженерія.**
ШІ може написати будь-який алгоритм на будь-якій мові програмування. Але **зрозуміти**, який саме алгоритм потрібен у конкретних умовах — це задача інженера. Вам потрібно або шукати алгоритм, який є одночасно стабільним і працює *на місці* (наприклад, Block Sort, що значно складніше), або зрозуміти, що класичні алгоритми в RAM тут не підійдуть, і застосувати зовнішнє сортування (**External Merge Sort**).

> **Як це працює без магії?** Ми читаємо лог з диску шматками, які гарантовано влазять у RAM (наприклад, по 100 МБ), сортуємо їх звичайним QuickSort чи `std::sort`, записуємо ці тимчасові відсортовані файли назад на диск, а потім лінійно зливаємо їх (Merge) в один фінальний файл без повного завантаження в пам'ять. Це показує, як складні інженерні задачі на гігабайтах даних вирішуються елегантною комбінацією найпростіших базових інструментів.

Теорія про $O(N \log N)$, in-place сортування та стабільність — це не просто застарілі питання для екзаменів. Це ваш інженерний словник, який дозволяє бачити обмеження системи, ставити правильні задачі штучному інтелекту та приймати архітектурні рішення.

### Архітектурний челендж: Як це вирішує Google (TrueTime)

Якщо замислитись ще глибше, сама вимога *"зберегти початковий порядок для подій в одну мілісекунду"* може бути архітектурно хибною, особливо якщо логи збираються з різних серверів. У розподілених системах "одна мілісекунда" — це величезна похибка, і єдиного "істинного часу" просто не існує.

Google у своїй глобальній базі даних **Spanner** зіткнувся з цією проблемою: як надійно впорядковувати (сортувати) транзакції з дата-центрів по всьому світу? 
Їхнє рішення — **TrueTime API**. У дата-центрах Google встановлені спеціальні GPS-приймачі та **атомні годинники**. Замість того, щоб видавати системі точну (але потенційно хибну) мітку часу, TrueTime повертає **гарантований інтервал часу** `[найраніший_можливий_час, найпізніший_можливий_час]` (похибка зазвичай складає кілька мілісекунд).

Як це змінює підхід до впорядкування подій?
1. **Інтервали не перетинаються:** Якщо часове вікно транзакції А закінчилося до початку вікна Б (`A.latest < B.earliest`), система на 100% впевнена, що А була раніше. Їх легко та безпечно сортувати.
2. **Інтервали перетинаються:** Це означає, що події відбулися *конкурентно* в межах похибки годинників. У такому разі їхній відносний порядок не має вирішального значення для причинно-наслідкових зв'язків.
3. **Commit Wait (Штучна затримка):** Щоб гарантувати лінійність і строгий порядок залежних транзакцій (щоб їхні вікна не перетнулися), Spanner просто змушує систему *почекати*, поки вікно похибки поточного часу не мине, перш ніж підтвердити успіх транзакції користувачу.

Отже, замість того, щоб шукати надскладний алгоритм стабільного сортування в умовах браку пам'яті, інженери Google **змінили саму парадигму часу**, перейшовши від мікросекундних точок до інтервалів та гарантій неперетину. Коли класичний алгоритмічний підхід заходить у глухий кут, справжня інженерія кидає виклик самим вхідним даним.

### Мікрокейс: Мислення поза шаблонами

Вам треба відсортувати масив з 10 мільйонів `bool` значень (тільки `true` та `false`). Який алгоритм сортування ви оберете і чому?

<details markdown="1">
<summary>Відповідь</summary>

**Жоден з класичних $O(N \log N)$ алгоритмів.** 
Оскільки можливих значень лише два, достатньо просто пройти масив один раз (за $O(N)$), порахувати кількість `false` (нехай їх $K$), а потім перезаписати перші $K$ елементів масиву як `false`, а решту — як `true`. Це абсолютно найшвидше та найефективніше рішення, яке є концептуальною основою для **Counting Sort** (сортування підрахунком). Це і є інженерне мислення: розуміти природу даних, а не сліпо застосовувати стандартні шаблони.

</details>

## Контрольні питання

1. Чому QuickSort називають "нестабільним"? Наведіть приклад, коли це важливо.

<details markdown="1">
<summary>Відповідь</summary>

QuickSort може переставляти рівні елементи під час partition. Приклад: сортуємо студентів за оцінкою. Якщо два студенти мають оцінку "5", але один здав роботу раніше, QuickSort може змінити їх порядок. Для збереження хронології треба використовувати MergeSort або `std::stable_sort`.

</details>

2. У вас є масив з 1 мільйоном елементів, але він майже відсортований (лише 100 елементів не на місці). Який алгоритм буде найшвидшим?

<details markdown="1">
<summary>Відповідь</summary>

**Insertion Sort** (O(N) для майже відсортованих даних) або **TimSort** (гібрид Insertion + Merge, використовується в Python та Java). `std::sort` (IntroSort) також швидко впорається, але Insertion Sort буде оптимальнішим для цього випадку.

</details>

3. Чому MergeSort вимагає O(N) додаткової пам'яті, а QuickSort — тільки O(log N)?

<details markdown="1">
<summary>Відповідь</summary>

**MergeSort** створює тимчасові масиви для злиття — розміром N.
**QuickSort** працює in-place (на місці), але використовує стек викликів для рекурсії. Глибина стеку — log N у середньому (висота дерева рекурсії).

</details>

4. Що станеться, якщо ви викличете `std::sort` на масиві, який вже відсортований?

<details markdown="1">
<summary>Відповідь</summary>

**IntroSort** (std::sort) впорається за O(N log N). Але для вже відсортованих даних оптимальніше використати `std::is_sorted` (O(N) перевірка), щоб уникнути зайвої роботи.

</details>
