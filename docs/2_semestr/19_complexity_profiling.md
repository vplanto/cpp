# Лекція 19: Performance Engineering — Big O та Брехня CPU

[← Лекція 18](18_modern_cpp_move_smart.md) | [Index](index.md) | [Далі: Лекція 20 →](20_lists_vector.md)


## Мета
Перестати гадати, чому код гальмує. Зрозуміти різницю між "Алгоритмічною складністю" (Big O) та "Реальною швидкістю" (Hardware Latency). Навчитися бачити O(N^2) до запуску коду.

## Експрес-опитування
1.  Що швидше: прочитати 1 МБ даних послідовно чи прочитати 1 МБ даних стрибками по 4 байти у випадкових місцях?
2.  Якщо алгоритм А робить N операцій, а алгоритм Б — 100 * N, чи однаково вони "масштабуються" (сповільнюються) при зростанні N?
3.  Якщо час роботи алгоритму взагалі не залежить від кількості даних, чи гарантує це миттєве виконання?

<details markdown="1">
<summary>Інженерна відповідь</summary>

1.  **Послідовно — у ~30 разів швидше.** Механізм **Prefetching** процесора передбачає ваші кроки і завантажує дані в кеш наперед (Cache Lines). Випадковий доступ (Linked List) ламає цю оптимізацію (Cache Misses).
2.  **Однакову — O(N).** В Big O ми ігноруємо константи та молодші члени. Але в реальному житті цей множник 100 може бути критичним.
3.  **Ні.** O(1) означає, що час **не залежить** від розміру вхідних даних N. Але ця "константа" може дорівнювати 5 хвилинам (наприклад, ініціалізація з'єднання з супутником).

</details>

## Залізо vs Математика: Чому секунди брешуть?

Перш ніж заглиблюватися в математику, дайте відповідь: **якщо алгоритм А працює 2 секунди, а алгоритм Б — 5 секунд, який з них кращий?**

На світанку програмування ми оцінювали алгоритми за **секундоміром**, вимірюючи фізичний час виконання на конкретному залізі. Але цей метод має "три великі брехні":
1.  **Брехня заліза:** На Intel i9 ваш код летить, на старому ноутбуці — повзе.
2.  **Брехня навантаження:** Якщо в фоні відкритий Chrome, результати будуть іншими.
3.  **Брехня вхідних даних:** На 10 елементах один код швидкий, на 10 мільйонах він може "вбити" сервер.

Комп'ютерним наукам потрібна була **Абстракція** — спосіб порівняти алгоритми незалежно від того, де вони запускаються.

### Від Секунд до Операцій
Ми перестали рахувати час і почали рахувати **Кроки** (елементарні операції: додавання, порівняння, копіювання).
- Нам не важливо, скільки *наносекунд* займає один крок на i9.
- Нам важливо, як **кількість цих кроків** змінюється залежно від розміру даних $N$.

Це і є **Big O Notation** — мова, яка описує не швидкість "заліза", а **масштабованість вашої ідеї**.

## Частина 1: Big O Notation (Теорія)

Ми оцінюємо алгоритми за тим, як **росте час виконання** при зростанні вхідних даних (N) у найгіршому випадку (Worst Case).

### Правила аналізу коду (Code Analysis Rules)

**Правило 1: Цикли (For Loops)**
Складність = Кількість ітерацій * Складність тіла циклу.
```cpp
// N ітерацій, тіло O(1)
for (int i = 0; i < N; i++) {
    k++; 
}
// Total: O(N)

```

**Правило 2: Вкладені цикли (Nested Loops)**
Складність = Добуток розмірів циклів.

```cpp
// Зовнішній: N разів
for (int i = 0; i < N; i++) {
    // Внутрішній: N разів
    for (int j = 0; j < N; j++) {
        k++; 
    }
}
// Total: O(N * N) = O(N^2)

```

**Правило 3: Послідовні дії**
Ми додаємо складності, але залишаємо тільки домінуючу (найбільшу).

```cpp
for (int i = 0; i < N; i++) { ... }       // O(N)
for (int i = 0; i < N; i++) {             // O(N^2)
   for (int j = 0; j < N; j++) { ... } 
}
// Math: O(N + N^2)
// Engineering: O(N^2) (Квадрат "з'їдає" лінію при великих N)

```

### Шкала жаху (від кращого до гіршого):

| Нотація | Назва | Приклад | Коментар |
| --- | --- | --- | --- |
| **O(1)** | Constant | `arr[i]`, додавання у `vector` | Мрія інженера. |
| **O(log N)** | Logarithmic | Binary Search, `std::map` | Дуже швидко. Подвоєння даних додає лише +1 крок. |
| **O(N)** | Linear | `for` loop, `std::find` | Норма. Час росте пропорційно даним. |
| **O(N log N)** | Linearithmic | `std::sort`, MergeSort | Найкращий час для сортування. |
| **O(N^2)** | Quadratic | Bubble Sort, Nested loops | **Гальмує** при N > 10,000. |
| **O(2^N)** | Exponential | Fibonacci Recursive | **Смерть** при N > 40. |

### Межа можливостей: Скільки даних обробить кожен алгоритм?

Щоб відчути різницю між складностями, ми виберемо єдину часову міру — **31.7 року** (умовна тривалість однієї кар'єри програміста, або мільярд секунд). Подивимося, який **максимальний обсяг даних ($N$)** зможе обробити кожен алгоритм за цей час:

| Алгоритм | Максимальний обсяг даних ($N$) за 31.7 року | Реальний орієнтир масштабу |
|-----------|------------------------------------------|-------------------|
| **O(n)** | $1,000,000,000,000,000,000$ ($10^{18}$) | Уся інформація в інтернеті |
| **O(n log n)** | $40,000,000,000,000,000$ | Потужний дата-центр |
| **O(n²)** | $1,000,000,000$ ($10^9$) | Одна велика таблиця в базі даних |
| **O(2ⁿ)** | **60 елементів** | Крихітний список паролів |

**Ключові висновки:**
- **Сила лінійності ($O(n)$):** Ви можете "перемолоти" майже всі дані світу за своє життя, і у вас ще залишиться час на каву.
- **Жах експоненти ($O(2^n)$):** Вам достатньо додати лише **60 елементів** (наприклад, у задачу про пошук шляху або злам пароля), щоб комп'ютер "завис" на цілу кар'єру програміста.
- **Висновок:** Перехід від $O(2^n)$ до $O(n^2)$ — це не просто "оптимізація", це перехід від "неможливого" до "буденного".

> **Інженерне правило:** Якщо ваш алгоритм O(2ⁿ) і n > 30, ви маєте проблему. Навіть найшвидший суперкомп'ютер не допоможе.

### Бонус: P, NP та Стіна "Неможливого"

Окрім математичної складності, програмісту важливо розуміти межі можливостей сучасної обчислювальної техніки:

*   **Клас P (Polynomial):** Задачі, які ми вміємо розв'язувати ефективно ($O(N^k)$). Сюди входить майже все, що ви пишете: сортування, пошук, робота з базами даних.
*   **Клас NP (Nondeterministic Polynomial):** Задачі, для яких ми не знаємо "швидкого" алгоритму пошуку (зазвичай це експоненціальні задачі $O(2^N)$), але можемо миттєво **перевірити** готову відповідь.
*   **NP-повні задачі (NP-Complete):** Найважчі задачі в класі NP, для яких час виконання зростає за експонентою ($O(2^N)$).

> **Важливо:** Якщо ви знайдете швидкий (поліноміальний) спосіб розв'язати **хоча б одну** NP-повну задачу, ви автоматично розв'яжете **всі** NP-повні задачі світу. Це відкриє шлях до зламу будь-яких шифрів та ідеальної логістики, але поки що ніхто не знає, чи це можливо ($P = NP?$).

**Приклади NP-повних "монстрів":**
1.  **Задача комівояжера (TSP):** Пошук ідеального (найкоротшого) маршруту між багатьма містами.
2.  **Задача про рюкзак:** Вибір найціннішого набору предметів при обмеженій вантажопідйомності.
3.  **Злам шифрів:** Пошук секретного ключа (вся сучасна криптографія тримається на тому, що це NP-важка задача).
4.  **Судоку:** Розв'язання узагальненого поля $N \times N$.

Якщо ваша робоча задача виявилася NP-повною — не намагайтеся знайти ідеальне рішення для великих $N$. Використовуйте **евристики** (наближені рішення).

## Частина 2: Hardware Reality (Масштаб часу)

Теорія алгоритмів часто ігнорує залізо. Це помилка.
Сучасний CPU набагато швидший за пам'ять (Memory Wall).

Щоб відчути це, уявімо, що **1 такт процесора (0.3 нс) = 1 секунда** людського життя.

| Подія | Реальний час | **Людський масштаб** (якщо 1 такт = 1 с) |
| --- | --- | --- |
| **1 CPU Cycle** | 0.3 ns | **1 секунда** (одне серцебиття) |
| **L1 Cache access** | 1 ns | **3 секунди** (взяти книгу зі столу) |
| **L2 Cache access** | 4 ns | **12 секунд** (дійти до книжкової полиці) |
| **RAM Access** (Main Memory) | 100 ns | **5 хвилин** (сходити за кавою в сусідню будівлю) |
| **SSD Access** (NVMe) | 100 us | **3-4 дні** (поїздка машиною Одеса-Львів і назад) |
| **HDD Access** (Rotational) | 10 ms | **1 рік** (піша подорож навколо світу) |
| **Internet Call** (Ping) | 150 ms | **15 років** (виростити дитину) |

> **Інсайт:** Кожного разу, коли процесор звертається до RAM (через `std::list` або Cache Miss), він "йде пити каву" на 5 хвилин. Ось чому структури, що тримають дані щільно (`std::vector`), перемагають.

## Частина 3: Практика — Benchmark (`std::chrono`)

Не вірте інтуїції ("мені здається, це швидко"). Міряйте.
Скопіюйте цей код і запустіть у себе.

```cpp
#include <iostream>
#include <vector>
#include <cmath>
#include <chrono> // 1. Підключаємо бібліотеку часу

// Тайпдеф для скорочення коду
using namespace std::chrono;

int main() {
    // Готуємо дані: 10 мільйонів чисел
    const int N = 10'000'000;
    std::vector<double> data(N, 1.5);

    std::cout << "Benchmarking " << N << " elements...\n\n";

    // --- Benchmark 1: Simple Sum (Легка операція) ---
    auto start = high_resolution_clock::now(); // 2. Старт
    
    double sum = 0;
    for(double val : data) {
        sum += val; // Процесор робить це за < 1 такту (суперскалярність)
    }

    auto stop = high_resolution_clock::now(); // 3. Стоп
    
    // 4. Переводимо в мікросекунди
    auto duration = duration_cast<microseconds>(stop - start);
    
    std::cout << "[Simple Loop]\n";
    std::cout << "Result: " << sum << "\n";
    std::cout << "Time:   " << duration.count() / 1000.0 << " ms\n"; // ~5-10 ms


    // --- Benchmark 2: Heavy Math (Важка операція) ---
    start = high_resolution_clock::now();
    
    double heavySum = 0;
    for(double val : data) {
        // Тригонометрія — це дуже дорого для CPU
        heavySum += std::sin(val) * std::cos(val);
    }

    stop = high_resolution_clock::now();
    duration = duration_cast<microseconds>(stop - start);
    
    std::cout << "\n[Heavy Loop]\n";
    std::cout << "Result: " << heavySum << "\n";
    std::cout << "Time:   " << duration.count() / 1000.0 << " ms\n"; // ~300-500 ms
    
    return 0;
}

```

**Приклад виводу в консолі:**

```bash
./test
Benchmarking 10000000 elements...

[Simple Loop]
Result: 1.5e+07
Time:   111.441 ms

[Heavy Loop]
Result: 705600
Time:   267.019 ms
```

> **Важливо:** Ці числа не є постійними. Якщо ви запустите цей код у себе, ви отримаєте інші значення. Час залежить від потужності вашого CPU, фонових процесів (наприклад, відкритий браузер) та навіть від того, чи підключений ноутбук до розетки (енергозбереження).

## Частина 4: Рекурсія — Друг чи Ворог?

Класичний приклад неефективності — рекурсивний Фібоначчі.

```cpp
// O(2^N) — Експоненціальна складність
long fib(int n) {
    if (n <= 1) return 1;
    return fib(n-1) + fib(n-2);
}

```

**Чому це погано?**

* Щоб порахувати `fib(5)`, ми рахуємо `fib(3)` **двічі**.
* Щоб порахувати `fib(50)`, комп'ютеру знадобляться роки.
* Дерево викликів росте як лавина.
* **Накладні витрати (Overhead):** Кожен рекурсивний виклик — це створення нового **Stack Frame** у пам'яті, копіювання параметрів та перемикання контексту на рівні процесора. Це набагато повільніше за звичайний цикл.

**Інженерне рішення:** Використовувати **цикл** (Iteration) або **Memoization** (запам'ятовування вже порахованих значень). У циклі це O(N).

---

> **Золоте правило:**
> **Структура даних = Спосіб зберігання в пам'яті = Алгоритм**
> **Алгоритм = Структура даних**

Це означає, що вибір того, **як** дані лежать у пам’яті, автоматично визначає межу швидкості вашого коду:

*   **Приклад 1 (Кеш-локальність):** У `std::vector` дані лежать одним суцільним блоком. Процесор читає їх "пачками" (Cache Lines). Тому звичайний цикл по вектору завжди переможе будь-який інший спосіб.
*   **Приклад 2 (Покажчики):** У `std::list` елементи розкидані по всій пам’яті. Щоб знайти наступний, процесор мусить чекати сотні тактів, поки дані прийдуть з RAM. Структура "зв’язаний список" сама по собі є повільним алгоритмом для сучасних CPU.
*   **Приклад 3 (Трейд-офф):** Хеш-таблиця використовує багато "зайвої" пам'яті, щоб перетворити складний пошук на миттєвий доступ. Ви буквально міняєте гігабайти пам'яті на мілісекунди часу.

## Ключові концепції лекції

| Концепція | Коротко: Для чого це? |
| :--- | :--- |
| **Big O Notation** | Математична абстракція для оцінки масштабованості алгоритму незалежно від заліза. |
| **Hardware Latency** | Розуміння реальних масштабів часу: CPU працює в секундах, RAM — у хвилинах, диск — у роках. |
| **Benchmarking** | Єдиний спосіб дізнатися правду: вимірювання часу виконання за допомогою `std::chrono`. |
| **P vs NP** | Межа між задачами, які можна вирішити швидко, та "монстрами", що потребують евристик. |
| **Overhead** | "Накладні витрати" (наприклад, на виклики рекурсії), які не враховує Big O, але відчуває CPU. |
| **Hidden Complexity** | Коли одна "проста" операція всередині циклу (наприклад, `insert`) непомітно робить код у 1000 разів повільнішим. |

## Контрольні питання

1. **Питання:** У вас є вкладений цикл: зовнішній йде до N, а внутрішній — фіксовано 100 разів. Яка складність: O(N^2) чи O(N)?
<details markdown="1">
<summary>Відповідь</summary>

**O(N).** Точний розрахунок: N * 100. В Big O константи відкидаються. Для великих N це лінійний алгоритм.
</details>

2. **Питання:** Чому доступ до жорсткого диска (HDD) називають "вбивцею продуктивності"?
<details markdown="1">
<summary>Відповідь</summary>

У "людському масштабі" доступ до HDD займає **1 рік** (порівняно з 1 секундою CPU). Будь-яка синхронна операція з диском у головному потоці заморожує програму (UI Lag).
</details>

3. **Питання:** Алгоритм A робить 1000 * N операцій, а Алгоритм B робить N^2. При якому (приблизно) N алгоритм B стане повільнішим за A?
<details markdown="1">
<summary>Відповідь</summary>

**При N > 1000.**
Рівняння: 1000*N = N^2 => N = 1000.
При малих даних (N < 1000) квадратичний алгоритм може бути швидшим через менші накладні витрати (overhead). Але ми оптимізуємо під великі дані, тому O(N) завжди виграє в перспективі.
</details>

4. **Питання:** Ви використовуєте `std::vector::insert` (вставка в початок) всередині циклу `for`. Яка загальна складність?
<details markdown="1">
<summary>Відповідь</summary>

**O(N^2).**
Цикл дає N. Вставка в початок вектора змушує перемістити всі існуючі елементи вправо — це операція O(N).
Разом: N * N = N^2. Це класична помилка продуктивності (Hidden Complexity). Для таких задач краще підходить `std::list` або `std::deque`.
</details>
