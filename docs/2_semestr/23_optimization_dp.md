## Мета

Зрозуміти, коли жадібні алгоритми (Greedy) працюють, а коли потрібне Dynamic Programming (DP). Навчитися розв'язувати задачу про рюкзак (Knapsack Problem). Розуміти принцип мемоїзації.

## Експрес-опитування

1. У вас є монети номіналом 1, 5, 10, 25 центів. Чи дасть жадібний алгоритм (який завжди обирає найбільшу монету) правильну відповідь, якщо потрібно видати здачу у 30 центів?
2. Чи можна розв'язати будь-яку оптимізаційну задачу жадібним алгоритмом?
3. Якщо під час обчислень ви знову й знову стикаєтеся з абсолютно однаковими підзадачами (наприклад, рахуєте `fib(3)` десять разів), яка очевидна ідея виникає, щоб не робити зайву роботу?

<details markdown="1">
<summary>Інженерна відповідь</summary>

1. **Так.** Жадібний алгоритм: 25 + 5 = 30 (2 монети). Це оптимально. Але для інших наборів монет (наприклад, 1, 3, 4) жадібний алгоритм може дати неоптимальну відповідь.
2. **Ні.** Greedy працює тільки тоді, коли задача має властивість "Greedy Choice Property" — локально оптимальний вибір веде до глобального оптимуму. Для задач, де потрібно врахувати майбутні наслідки, потрібен DP.
3. **Запам'ятати (записати) результат першого обчислення,** щоб потім просто брати готове значення. Цей підхід називається мемоїзацією (memoization) і є основою динамічного програмування (DP).

</details>

## Частина 1: Greedy Algorithms — коли жадібність працює

**Ідея:** На кожному кроці робити локально найкращий вибір, сподіваючись, що це призведе до глобального оптимуму.

### Приклад 1: Coin Change (для певних наборів монет)

Задача: Дати здачу N центів мінімальною кількістю монет.

Набір монет: `{1, 5, 10, 25}` (центи США).

```cpp
#include <vector>
#include <iostream>

int greedyCoinChange(int amount, const std::vector<int>& coins) {
    int count = 0;
    
    // Монети повинні бути відсортовані за спаданням
    for (int coin : coins) {
        while (amount >= coin) {
            amount -= coin;
            count++;
            std::cout << coin << " ";
        }
    }
    
    std::cout << "\\nTotal coins: " << count << "\\n";
    return count;
}

int main() {
    std::vector<int> coins = {25, 10, 5, 1};
    greedyCoinChange(63, coins); // Виведе: 25 25 10 1 1 1 (6 монет)
}
```

<details markdown="1">
<summary><b>Контрольне питання:</b> Яка часова складність цього алгоритму та як її оптимізувати?</summary>

У поточній реалізації з циклом `while (amount >= coin)` часова складність у найгіршому випадку становить **O(amount / min_coin)**, що для великих значень здачі (наприклад, мільйон одиничних монет) може працювати повільно. Також можна сказати, що складність дорівнює **O(N + C)**, де **N** — кількість номіналів монет, а **C** — загальна кількість виданих монет.

**Оптимізація:**
Ми можемо позбутися циклу `while`, використовуючи ділення (`/`) та залишок від ділення (`%`). Тоді алгоритм працюватиме за лінійний час **O(N)** відносно кількості номіналів монет:

```cpp
int optimizedGreedyCoinChange(int amount, const std::vector<int>& coins) {
    int count = 0;
    for (int coin : coins) {
        int numCoins = amount / coin; // Скільки таких монет вміщується
        count += numCoins;
        amount %= coin;              // Залишок здачі
        if (numCoins > 0) {
            std::cout << numCoins << "x" << coin << " ";
        }
    }
    std::cout << "\nTotal coins: " << count << "\n";
    return count;
}
```
</details>

**Чому це працює для {1, 5, 10, 25}?**
Ці номінали спеціально підібрані так, що жадібний вибір завжди оптимальний. Це називається **Canonical Coin System**.

### Приклад 2: Activity Selection (вибір максимальної кількості занять)

Задача: У вас є N занять з часом початку та кінця. Виберіть максимальну кількість занять, які не перетинаються.

**Greedy Strategy:** Завжди вибирайте заняття, яке **закінчується найраніше**.

```cpp
#include <vector>
#include <algorithm>

struct Activity {
    int start, finish;
};

int activitySelection(std::vector<Activity>& activities) {
    // Сортуємо за часом закінчення
    std::sort(activities.begin(), activities.end(), 
              [](const Activity& a, const Activity& b) {
                  return a.finish < b.finish;
              });
    
    int count = 1;
    int lastFinish = activities[0].finish;
    
    for (int i = 1; i < activities.size(); i++) {
        if (activities[i].start >= lastFinish) {
            count++;
            lastFinish = activities[i].finish;
        }
    }
    
    return count;
}
```

**Складність алгоритму:**
- **Часова складність:** **O(N log N)**, де **N** — кількість занять. Сортування за допомогою `std::sort` займає **O(N log N)** часу, а наступний прохід циклом — лінійний час **O(N)**.
- **Просторова складність:** **O(1)** додаткової пам'яті (якщо сортування виконується in-place; інакше **O(log N)** додаткової пам'яті для стека викликів алгоритму швидкого сортування).

**Чому це працює?**
Вибираючи заняття, яке закінчується найраніше, ми залишаємо максимум часу для наступних занять.

## Частина 2: Коли Greedy НЕ працює

### Антиприклад: Coin Change для {1, 3, 4}

Дати здачу 6 центів.

- **Greedy:** 4 + 1 + 1 = **3 монети** (suboptimal).
- **Optimal:** 3 + 3 = **2 монети**.

Жадібний алгоритм вибрав 4 (найбільшу), але це не призвело до оптимуму.

**Висновок:** Greedy працює не завжди. Для таких задач потрібен **Dynamic Programming**.

## Частина 3: Dynamic Programming — ідея

DP розбиває задачу на **підзадачі**, розв'язує їх один раз і **запам'ятовує результати** (memoization).

### Принципи DP

1. **Optimal Substructure:** Оптимальне рішення задачі складається з оптимальних рішень підзадач.
2. **Overlapping Subproblems:** Підзадачі повторюються багато разів.

### Приклад: Fibonacci (Naive vs DP)

**Naive Recursion (O(2^N)):**
```cpp
int fib(int n) {
    if (n <= 1) return n;
    return fib(n-1) + fib(n-2); // Рахує fib(5) багато разів
}
```

**DP Solution (O(N)):**
```cpp
#include <vector>

int fibDP(int n) {
    if (n <= 1) return n;
    
    std::vector<int> dp(n + 1);
    dp[0] = 0;
    dp[1] = 1;
    
    for (int i = 2; i <= n; i++) {
        dp[i] = dp[i-1] + dp[i-2]; // Використовуємо вже обчислені значення
    }
    
    return dp[n];
}
```

**Складність алгоритму:**
- **Часова складність:** **O(N)** — виконується один цикл від `2` до `N`, де кожна ітерація виконує одну арифметичну операцію за **O(1)**.
- **Просторова складність:** **O(N)** — через виділення додаткового вектора розміром `N + 1`.

**Оптимізація пам'яті до O(1):**
Оскільки для обчислення `dp[i]` нам потрібні лише два попередні значення (`dp[i-1]` та `dp[i-2]`), нам не обов'язково зберігати весь масив. Ми можемо використовувати лише дві змінні:

```cpp
int fibOptimized(int n) {
    if (n <= 1) return n;
    
    int prev2 = 0; // fib(0)
    int prev1 = 1; // fib(1)
    int current = 0;
    
    for (int i = 2; i <= n; i++) {
        current = prev1 + prev2;
        prev2 = prev1;
        prev1 = current;
    }
    
    return current;
}
```
Ця версія має часову складність **O(N)** та просторову складність **O(1)** додаткової пам'яті.

**Ключова ідея:** Замість того, щоб рахувати `fib(5)` кілька разів, ми зберігаємо його у `dp[5]` і використовуємо повторно.

## Частина 4: Knapsack Problem — класика DP

Задача: У вас є рюкзак з максимальною вагою W. Є N предметів, кожен має вагу `w[i]` та цінність `v[i]`. Виберіть предмети так, щоб сумарна цінність була максимальною, але вага не перевищувала W.

### 0/1 Knapsack (кожен предмет можна взяти лише раз)

**Чому "0/1"?** Бо кожен предмет можна або взагалі не брати (**0**), або взяти повністю в одному екземплярі (**1**). Ми не можемо відрізати частину предмета (наприклад, половину телевізора) чи брати один і той самий предмет декілька разів.

#### Візуалізація та покроковий приклад

У коді таблиця `dp` має розмір `(N + 1) x (W + 1)`.
Нехай місткість рюкзака `W = 5`, і у нас є `N = 3` предмети:
1. **Спальник** (Предмет 1): вага = 2, цінність = 3
2. **Ноутбук** (Предмет 2): вага = 3, цінність = 4
3. **Намет** (Предмет 3): вага = 4, цінність = 5

Таблиця `dp[i][w]` заповнюється зліва направо, згори вниз:

| `i` (предмети) \ `w` (вага) | 0 | 1 | 2 | 3 | 4 | 5 |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **0** (порожній рюкзак) | 0 | 0 | 0 | 0 | 0 | 0 |
| **1** (тільки **Спальник**: в=2, ц=3) | 0 | 0 | **3** | 3 | 3 | 3 |
| **2** (Спальник + **Ноутбук**: в=3, ц=4) | 0 | 0 | 3 | 4 | 4 | **7** |
| **3** (Спальник + Ноутбук + **Намет**: в=4, ц=5) | 0 | 0 | 3 | 4 | 5 | **7** |

**Як обчислюється рядок 1 (тільки Спальник: в=2, ц=3):**
Для кожної ваги `w` від 0 до 5 ми визначаємо, чи можемо вмістити Спальник:
* Якщо `w < 2` (для `w = 0` та `w = 1`): Спальник має вагу 2 кг, тому він не вміщується. Значення копіюється з попереднього рядка: `dp[1][w] = dp[0][w] = 0`.
* Якщо `w >= 2` (для `w = 2, 3, 4, 5`): Спальник вміщується. Вибір стоїть між:
  1. Не брати його: цінність = `dp[0][w] = 0`.
  2. Взяти його: цінність = `dp[0][w - 2] + 3 = 0 + 3 = 3`.
  Обираємо максимум: `max(0, 3) = 3`.
Отже, весь рядок 1 набуває значень: `[0, 0, 3, 3, 3, 3]`.

**Як обчислюється клітинка `dp[2][5]` (додаємо Ноутбук при ліміті ваги 5):**
Формула: `dp[2][5] = max(dp[1][5], dp[1][5 - 3] + 4)`

1. **Варіант 1 (НЕ беремо Ноутбук):** Дивимося на значення безпосередньо над поточною клітинкою (`dp[1][5]`, тобто найкраща цінність без Ноутбука). Отримуємо **3**.
2. **Варіант 2 (Беремо Ноутбук):** Забираємо під нього вагу `w_2 = 3` (залишається `5 - 3 = 2`). Дивимося в попередньому рядку найкращу цінність для вільної ваги 2 (`dp[1][2] = 3`, тобто цінність Спальника) і додаємо до неї цінність поточного предмета (`v_2 = 4`, цінність Ноутбука). Отримуємо `3 + 4 =` **7**.
3. **Обираємо максимум:** `max(3, 7) = 7`. Записуємо 7 у `dp[2][5]`.

**DP Formula (Формула переходу):**
Для кожного предмета `i` (з вагою `w_i` та цінністю `v_i`) ми маємо два варіанти:

```
dp[i][w] = max(
    dp[i-1][w],             // Варіант 1: НЕ беремо предмет i (результат такий же, як для i-1 предметів)
    dp[i-1][w - w_i] + v_i  // Варіант 2: Беремо предмет i (додаємо цінність v_i, а залишок ваги w - w_i беремо з попереднього кроку)
)
```

* `dp[i][w]` — максимальна цінність для перших `i` предметів при доступній вазі рюкзака `w`.
* `dp[i-1][w]` — найкращий результат без поточного предмета.
* `dp[i-1][w - w_i] + v_i` — найкращий результат, якщо ми виділили місце під предмет `i` (зменшили доступну вагу на `w_i`) і додали його цінність `v_i`.

**Код:**
```cpp
#include <vector>
#include <algorithm>

int knapsack(int W, const std::vector<int>& weights, const std::vector<int>& values) {
    int n = values.size();
    std::vector<std::vector<int>> dp(n + 1, std::vector<int>(W + 1, 0));
    
    for (int i = 1; i <= n; i++) {
        for (int w = 0; w <= W; w++) {
            // Не беремо предмет i
            dp[i][w] = dp[i-1][w];
            
            // Намагаємося взяти предмет i
            if (weights[i-1] <= w) {
                dp[i][w] = std::max(dp[i][w], 
                                    dp[i-1][w - weights[i-1]] + values[i-1]);
            }
        }
    }
    
    return dp[n][W];
}

int main() {
    std::vector<int> weights = {2, 3, 4};
    std::vector<int> values  = {3, 4, 5};
    int W = 5;
    
    std::cout << "Max value: " << knapsack(W, weights, values) << "\n"; // 7
}
```

**Результат роботи:**
```text
Max value: 7
```
(Для рюкзака місткістю `W = 5` оптимальним рішенням є вибір предметів **Спальник** та **Ноутбук** із загальною цінністю 7).

**Складність:** O(N * W) — де N = кількість предметів, W = максимальна вага.

## Частина 5: Greedy vs DP — коли що використовувати

| Характеристика | Greedy | Dynamic Programming |
|----------------|--------|---------------------|
| **Складність** | Зазвичай O(N log N) | Зазвичай O(N^2) або O(N * M) |
| **Оптимальність** | Не завжди гарантована | Завжди оптимальне рішення |
| **Пам'ять** | O(1) або O(N) | O(N * M) — таблиця мемоїзації |
| **Приклади де працює Greedy** | Activity Selection, Huffman Coding, Dijkstra | — |
| **Приклади де потрібен DP** | — | Knapsack, Longest Common Subsequence, Coin Change (загальний випадок) |

**Правило:** Спочатку спробуйте Greedy (простіше і швидше). Якщо не працює — переходьте до DP.

## Бонус: Union-Find та Алгоритм Крускала (Advanced)

<details markdown="1">
<summary><b>Розгорнути для вивчення теми Union-Find та MST</b></summary>

**Union-Find (Disjoint Set Union, DSU)** — структура даних для відстеження розбиття множини на **непересічні підмножини**. Підтримує дві ефективні операції:

- **`find(x)`** — знайти представника (корінь) множини, до якої належить `x`
- **`union(x, y)`** — об'єднати множини `x` та `y` в одну

### Задача, яку вирішує Union-Find

```
Є 6 міст. З'єднуємо їх поступово дорогами:
  union(0,1)  → {0,1} {2} {3} {4} {5}
  union(2,3)  → {0,1} {2,3} {4} {5}
  union(1,3)  → {0,1,2,3} {4} {5}

Питання: "Чи з'єднані місто 0 і місто 2?" → find(0) == find(2) ? → ТАК
Питання: "Чи з'єднані місто 0 і місто 4?" → find(0) == find(4) ? → НІ
```

> 💡 **Чому дорівнює `find(0)` і як це працює під капотом?**
>
> На початку кожен елемент є представником самого себе (`parent[i] = i`).
> 1. Після `union(0, 1)`: корінь `0` вказує на `1`. `parent[0] = 1`.
> 2. Після `union(2, 3)`: корінь `2` вказує на `3`. `parent[2] = 3`.
> 3. Після `union(1, 3)`: корінь `1` (що дорівнює `1`) стає піддеревом кореня `3` (що дорівнює `3`). `parent[1] = 3`.
>
> Тепер під час пошуку представника для `0` через `find(0)` ми проходимо ланцюжок зв'язків:
> * `find(0)` дивиться на `parent[0]`, який дорівнює `1`.
> * Далі йде до `parent[1]`, який дорівнює `3`.
> * Оскільки `parent[3] == 3` (3 є власним коренем), рекурсія зупиняється і повертає **3**. Отже, **`find(0) == 3`**.
>
> Для міста `2` виклик `find(2)` відразу переходить до `parent[2]`, що дорівнює **3**. Отже, **`find(2) == 3`**.
>
> Оскільки **`find(0) == find(2)`** (обидва дають **3**), ми розуміємо, що вони належать до однієї підмножини (з'єднані). Для міста `4` значення `find(4)` дорівнює `4` (окреме дерево), тому `find(0) == find(4)` (тобто `3 == 4`) поверне `false`.

### Наївна реалізація

```cpp
struct UnionFind {
    std::vector<int> parent;

    UnionFind(int n) : parent(n) {
        // Спочатку кожен елемент — сам собі корінь
        std::iota(parent.begin(), parent.end(), 0); // {0,1,2,3,...}
    }

    int find(int x) {
        if (parent[x] == x) return x;
        return find(parent[x]); // Рекурсивно піднімаємось до кореня
    }

    void unite(int x, int y) {
        parent[find(x)] = find(y); // Корінь x стає дитиною кореня y
    }

    bool connected(int x, int y) {
        return find(x) == find(y);
    }
};
```

### Оптимізована реалізація (Path Compression + Union by Rank)

Без оптимізацій `find()` може деградувати до O(n). З двома трюками — амортизовано O(α(n)) ≈ O(1):

```cpp
struct UnionFind {
    std::vector<int> parent, rank;

    UnionFind(int n) : parent(n), rank(n, 0) {
        std::iota(parent.begin(), parent.end(), 0);
    }

    // Path Compression: на шляху від x до кореня всі вузли
    // підключаємо безпосередньо до кореня → дерево стає плоским
    int find(int x) {
        if (parent[x] != x)
            parent[x] = find(parent[x]); // Стискаємо шлях
        return parent[x];
    }

    // Union by Rank: менше дерево приєднуємо під більше
    void unite(int x, int y) {
        int rx = find(x), ry = find(y);
        if (rx == ry) return; // Вже в одній множині

        if (rank[rx] < rank[ry]) std::swap(rx, ry);
        parent[ry] = rx;                     // ry стає дитиною rx
        if (rank[rx] == rank[ry]) rank[rx]++; // Рівні ранги → збільшуємо
    }

    bool connected(int x, int y) { return find(x) == find(y); }
};

// Використання:
UnionFind uf(6);
uf.unite(0, 1);
uf.unite(2, 3);
uf.unite(1, 3);

std::cout << uf.connected(0, 2); // 1 (true) — в одній компоненті
std::cout << uf.connected(0, 4); // 0 (false) — різні компоненти
```

**Складність з оптимізаціями:** O(α(n)) на операцію, де α — **обернена функція Аккермана** (практично константа, ≤ 4 для будь-яких реальних n).

### Класичне застосування: Алгоритм Крускала (MST)

**MST (Minimum Spanning Tree / Мінімальне остове дерево)** — це підграф зв'язного зваженого неорієнтованого графа, який об'єднує всі його вершини без утворення циклів (остов) та має мінімально можливу сумарну вагу ребер.

*Приклад з життя:* Прокладання мережевого кабелю між корпусами університету так, щоб усі будівлі були з'єднані в мережу, але загальна довжина кабелю була мінімальною.

**Алгоритм Крускала** будує MST жадібно:
1. Сортує всі ребра графа за зростанням їхньої ваги.
2. Послідовно перевіряє кожне ребро: якщо воно з'єднує вершини з різних компонент зв'язності (тобто не утворює цикл), ми додаємо його до остова та об'єднуємо ці компоненти.
3. Перевірка на цикли та об'єднання виконуються миттєво завдяки **Union-Find** (операції `connected` та `unite`).

```cpp
// Мінімальне остове дерево (Minimum Spanning Tree)
struct Edge { int u, v, w; };

int kruskal(int n, std::vector<Edge>& edges) {
    std::sort(edges.begin(), edges.end(),
              [](const Edge& a, const Edge& b) { return a.w < b.w; });

    UnionFind uf(n);
    int totalWeight = 0;

    for (auto& [u, v, w] : edges) {
        if (!uf.connected(u, v)) { // Не утворюємо цикл
            uf.unite(u, v);
            totalWeight += w;
        }
    }
    return totalWeight;
}
```

**Типові задачі для Union-Find:**
- Кількість компонент зв'язності у динамічному графі
- Визначення, чи утворює додавання ребра цикл
- Мінімальне остове дерево (Kruskal)
- Задачі "острови" на сітці

</details>

## Контрольні питання

1. Чому рекурсивний Fibonacci має складність O(2^N)?

<details markdown="1">
<summary>Відповідь</summary>

Кожен виклик `fib(n)` породжує 2 виклики (`fib(n-1)` та `fib(n-2)`). Це створює бінарне дерево викликів висотою N, де кількість вузлів ~2^N. Більшість викликів повторюються (наприклад, `fib(5)` обчислюється багато разів), але без мемоїзації ми рахуємо їх заново.

</details>

2. У чому різниця між 0/1 Knapsack та Fractional Knapsack?

<details markdown="1">
<summary>Відповідь</summary>

- **0/1 Knapsack:** Предмет можна взяти повністю або не брати взагалі. Потрібен DP (O(N*W)).
- **Fractional Knapsack:** Можна брати частини предмета. Тут працює Greedy (сортуємо за value/weight і беремо найвигідніші) — O(N log N).

</details>

3. Як визначити, чи можна розв'язати задачу Greedy алгоритмом?

<details markdown="1">
<summary>Відповідь</summary>

Перевірте дві властивості:
1. **Greedy Choice Property:** Локально оптимальний вибір веде до глобального оптимуму.
2. **Optimal Substructure:** Оптимальне рішення включає оптимальні рішення підзадач.

Якщо обидві виконуються — Greedy може спрацювати. Але довести це математично буває складно, тому найкраще — протестувати на кількох прикладах.

</details>

4. Чому таблиця DP у Knapsack має розмір `(N+1) x (W+1)`?

<details markdown="1">
<summary>Відповідь</summary>

- **N+1 рядків:** Перший рядок `dp[0][...]` = базовий випадок (0 предметів = 0 цінності).
- **W+1 стовпців:** Перший стовпець `dp[...][0]` = базовий випадок (вага рюкзака = 0 → не можемо нічого взяти).

Це дозволяє уникнути перевірок на межі масиву і спрощує логіку.

</details>
