# Лекція 16а: Поліморфізм, V-Table та Абстрактні класи

## Мета
Після цієї лекції ви зможете:
*   **Розуміти поліморфізм** — як одна і та ж функція поводиться по-різному залежно від типу об'єкта.
*   **Пояснити V-Table** — як працює динамічна диспетчеризація "під капотом" і скільки вона коштує в байтах та тактах.
*   **Створювати абстрактні класи** — проєктувати інтерфейси-контракти для гнучкої архітектури.
*   **Уникати витоків пам'яті** — знати, чому деструктор мусить бути віртуальним.

## Частина 1: Поліморфізм та Динамічне зв'язування

**Поліморфізм** (від грец. "багато форм") — це здатність одного і того ж виклику робити різні речі залежно від того, хто його виконує.

**Побутова аналогія:** Уявіть кнопку "Відтворити" (▶) на пульті. Ви натискаєте одну і ту ж кнопку, але:
*   Якщо пульт направлений на **телевізор** — починається фільм.
*   Якщо на **колонку** — грає музика.
*   Якщо на **проєктор** — світиться презентація.

Команда одна (`play()`), а поведінка різна — залежно від пристрою. В ООП "пристрій" — це тип об'єкта, а "кнопка" — виклик методу через вказівник на базовий клас. Але за замовчуванням C++ цього не вміє — давайте переконаємось.

### Крок 1: Створюємо проблему

Уявіть гру з різними типами юнітів. Ви створюєте базовий клас та нащадника:

```cpp
class Unit {
public:
    void move() { cout << "Unit: generic move" << endl; }
};

class Soldier : public Unit {
public:
    void move() { cout << "Soldier: tactical advance" << endl; }
};
```

Тепер ви хочете зберігати різних юнітів в одному масиві (через вказівник на базовий клас — це ж логічно, `Soldier` **є** `Unit`):

```cpp
Unit* squad[3];
squad[0] = new Soldier();
squad[1] = new Soldier();
squad[2] = new Unit();

for (int i = 0; i < 3; i++) {
    squad[i]->move();  // Що виведеться?
}
```

**Очікуємо:**
```text
Soldier: tactical advance
Soldier: tactical advance
Unit: generic move
```

**Отримуємо:**
```text
Unit: generic move
Unit: generic move
Unit: generic move
```

Всі три юніти рухаються як базовий `Unit`, навіть ті, що є `Soldier`. Солдати "забули", хто вони.

### Крок 2: Чому так? (Static Binding)

Компілятор дивиться на **тип вказівника** (`Unit*`), а не на **тип об'єкта** (`Soldier`). Він статично (на етапі компіляції) вирішує: "Тип — `Unit*`, отже викликаю `Unit::move()`". Це називається **Static Binding** — швидко, але "сліпо".

### Крок 3: Виправляємо — ключове слово `virtual`

Щоб компілятор визначав правильну версію методу **під час виконання** програми (а не під час компіляції), ми позначаємо метод батька як `virtual`:

```cpp
class Unit {
public:
    virtual void move() { cout << "Unit: generic move" << endl; }
};

class Soldier : public Unit {
public:
    void move() override { cout << "Soldier: tactical advance" << endl; }
};
```

> **Приклад в коді:** [oop.cpp](https://github.com/vplanto/cpp/blob/main/09-oop-intro/oop.cpp) (демонструє використання `virtual` та `override`, а також важливий нюанс — небезпеку рекурсії під час викликів).

Тепер той самий код з масивом виведе правильно:

```text
Soldier: tactical advance
Soldier: tactical advance
Unit: generic move
```

**Що змінилося:**
*   `virtual` у батька — каже компілятору: "Не вирішуй зараз, яку версію викликати. Переклади це рішення на рантайм."
*   `override` у нащадка — необов'язкове, але захищає від друкарських помилок (наприклад, `Move()` замість `move()` — компілятор одразу скаже про помилку).

## Частина 2: Внутрішній механізм — V-Table

Як програма дізнається, яку функцію викликати, якщо ми працюємо через вказівник на базовий клас?

Коли в класі з'являється хоча б одна `virtual` функція, компілятор створює:
1. **V-Table** — приховану таблицю (масив вказівників на функції) для кожного класу.
2. **vptr** — прихований вказівник всередині кожного об'єкта, який вказує на відповідну V-Table.

**Ціна поліморфізму (Trade-offs):**
* **Пам'ять:** Кожен об'єкт стає більшим на 8 байт (через vptr) на 64-бітних системах.
* **Швидкість:** Замість прямого виклику функції, процесор робить "підвійний стрибок" (спочатку в vptr, потім за адресою з таблиці).

## Частина 3: Абстрактні класи (Інтерфейси) та Контракти

Іноді нам потрібен клас лише як "контракт" або шаблон поведінки. Наприклад, ми не можемо створити "просто фігуру" (`Shape`), ми можемо створити тільки конкретне Коло чи Квадрат.

```cpp
class Shape {
public:
    // Pure Virtual Function (= 0)
    virtual double getArea() const = 0; 
};
```

* **Pure Virtual Function (Чиста віртуальна функція):** Це метод із записом `= 0` в кінці. Він **не має реалізації** в базовому класі. Це просто обіцянка: "Усі мої нащадки точно вмітимуть це робити".
* **Абстрактний клас:** Клас, що містить хоча б одну чисту віртуальну функцію.
* **Заборона:** Ви не можете створити об'єкт цього класу (`new Shape()` — помилка компіляції), бо він ще "недописаний".
* **Обов'язок (Контракт):** Нащадки мусять реалізувати всі такі методи. Якщо `Circle` наслідується від `Shape` і не реалізує `getArea()`, `Circle` теж стає абстрактним класом і його не можна створити.

**Чому сучасне програмування йде у бік інтерфейсів?**
Спадкування реалізації (з готовим кодом) часто призводить до сильної зв'язності (High Coupling) — пам'ятаєте "Крихкий базовий клас"? Сучасні архітектури (як SOLID) рекомендують **наслідувати інтерфейси**, а не реалізацію.
Інтерфейс у C++ — це абстрактний клас, де *всі* методи є чистими віртуальними (`= 0`) і немає жодних полів даних. Це створює **слабку зв'язність** (Low Coupling): класи спілкуються лише через "контракти" поведінки, не знаючи, як воно працює всередині.

## Частина 4: Віртуальний деструктор (Важливо для пам'яті)

Це критичний момент при роботі з поліморфізмом. Якщо ви видаляєте об'єкт нащадка через вказівник на базу, ви можете отримати витік пам'яті.

```cpp
class Base { 
    virtual ~Base() { } // ПОВИНЕН бути віртуальним
};

class Derived : public Base {
    int* data = new int[100]; 
    ~Derived() { delete[] data; } 
};

// ...
Base* ptr = new Derived();
delete ptr; // Викличеться ~Derived() ТІЛЬКИ якщо деструктор у Base віртуальний
```

> [!CAUTION]
> **Золоте правило:** Якщо у вашому класі є хоча б одна `virtual` функція — деструктор теж **мусить** бути віртуальним. Без цього `delete base_ptr` не викличе деструктор нащадка, і ресурси (пам'ять, файли) не звільняться.

## Частина 5: Множинне спадкування та Diamond Problem

C++ — одна з небагатьох мов, яка дозволяє наслідуватися від кількох класів одночасно.

```cpp
class Printer { void print(); };
class Scanner { void scan(); };
class MFP : public Printer, public Scanner { }; // Багатофункціональний пристрій
```

Але це породжує **Ромбоподібне спадкування (Diamond Problem)**:

```text
       Device (має поле isOn)
      /      \
 Printer    Scanner
      \      /
        MFP
```

Якщо `Printer` і `Scanner` наслідуються від `Device`, то `MFP` успадкує **дві копії** базового `Device`! У `MFP` буде два незалежних поля `isOn`. Якщо викличете `mfp.isOn`, компілятор видасть помилку: "Яку саме половину пристрою ви маєте на увазі?".

Щоб цього уникнути, використовують **віртуальне спадкування** (`class Printer : virtual public Device`), але це складно, повільно і заплутано.
> **Сучасний стандарт:** Уникайте множинного спадкування стану (класів з полями). Дозволяється лише множинне спадкування **інтерфейсів** (чистих абстрактних класів без полів), бо вони не дублюють дані в пам'яті.

## Частина 6: Підсумок зв'язків та Access Modifiers

### Таблиця: Спадкування, Композиція, Агрегація

Шпаргалка, щоб швидко обрати правильний зв'язок між класами:

| Тип зв'язку | Відношення | Хто керує життям? | Зв'язність (Coupling) | Приклад у коді |
|-------------|------------|------------------|------------------------|----------------|
| **Спадкування** | **Is-A** (Є чимось) | Батько і нащадок нероздільні | **Найсильніша** (Уникати, якщо можна) | `class Cat : public Animal` |
| **Композиція** | **Has-A** (Має / Частина) | Контейнер знищується → частина знищується | **Слабка** (Рекомендована) | `class Car { Engine e; };` |
| **Агрегація** | **Uses-A** (Використовує) | Частина живе незалежно від контейнера | **Найслабша** | `class Car { Driver* d; };` |

### Доступність (Access Modifiers) при різних зв'язках

Як працюють `private`, `protected`, `public` залежно від архітектури:

*   **Спадкування (Inheritance):** Нащадок бачить `public` та `protected` батька. `private` поля батька є в пам'яті нащадка, але код нащадка не має до них прямого доступу.
*   **Композиція / Агрегація (Composition / Aggregation):** Клас-"власник" може звертатися **тільки до `public`** методів об'єкта, яким він володіє. Він діє як звичайний зовнішній "клієнт".

> **Золоте правило інкапсуляції:** Ніколи не робіть поля даних `protected` або `public`. Поля завжди мають бути `private`. Якщо нащадку чи власнику потрібен доступ до даних, робіть це безпечно через `public` / `protected` методи (гетери/сетери).

## Бонус: Залишок принципів SOLID (S, I, D)

У попередній лекції ми розглянули Liskov Substitution (L) та Open/Closed (O). Тема абстрактних класів та поліморфізму дозволяє зрозуміти ще три фундаментальні ідеї:

**S — Single Responsibility Principle (Принцип єдиної відповідальності)**

Кожен клас (і кожен інтерфейс) повинен мати лише одну причину для зміни. Якщо ваш клас `Manager` одночасно рахує зарплату, підключається до бази даних і відправляє email — він порушує SRP і дуже скоро перетвориться на Spaghetti Code. Розділяйте логіку.

**I — Interface Segregation Principle (Принцип розділення інтерфейсу)**

Краще багато маленьких спеціалізованих інтерфейсів, ніж один "жирний" загальний.
*Помилка:* Створити `class IWorker { virtual void code()=0; virtual void sell()=0; };`. Програміст не вміє продавати, а менеджер не вміє кодити, але обидва змушені будуть малювати "пусті" методи (dummy overrides).
*Рішення:* Створити `iCoder` та `iSeller`. Класи реалізують лише те, що їм дійсно потрібно.

**D — Dependency Inversion Principle (Принцип інверсії залежностей)**

Верхні рівні архітектури не повинні залежати від нижніх. **Обидва мають залежати від абстракцій.**
*Помилка:* Клас `Game` (високий рівень) створює всередині себе об'єкт `OpenGLRenderer` (низький рівень). Щоб змінити графіку на DirectX, треба переписати `Game`.
*Рішення:* Створити абстрактний клас `IRenderer`. `Game` приймає вказівник на `IRenderer*` (через конструктор) і викликає `draw()`. Тепер `Game` не знає ні про OpenGL, ні про DirectX — він працює лише з *контрактом*. Це і є магія поліморфізму!

## Контрольні питання

1. Чому спадкування збільшує Coupling?
2. Що таке `override` і чому його варто писати, хоча це необов'язково?
3. Якщо `sizeof(int) = 4`, який розмір буде у класу:
```cpp
class A { 
    int x; 
    virtual void f(); 
};
```
(Підказка: згадайте про vptr і вирівнювання).
4. Чи може конструктор бути віртуальним? (Відповідь: Ні, бо vptr ще не ініціалізований).

<details markdown="1">
<summary>Відповіді</summary>

1. Бо дочірній клас залежить від реалізації батьківського. Зміни в батьку можуть зламати нащадка.
2. `override` просить компілятор перевірити, чи існує такий віртуальний метод у батька. Це захищає від помилок у назвах (наприклад, `move()` vs `Move()`).
3. **16 байт.** `int` (4 байта) + `vptr` (8 байт на 64-bit). Через вирівнювання (padding) компілятор додає пустоту, щоб vptr був за адресою, кратною 8.
4. **Ні.** Щоб викликати віртуальну функцію, потрібен vptr. Vptr записується в об'єкт саме під час роботи конструктора. До цього моменту поліморфізм не працює.

</details>
