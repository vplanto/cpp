# Лекція 16а: Поліморфізм, V-Table та Абстрактні класи

## Мета
Після цієї лекції ви зможете:
*   **Розуміти поліморфізм** — як одна і та ж функція поводиться по-різному залежно від типу об'єкта.
*   **Пояснити V-Table** — як працює динамічна диспетчеризація "під капотом" і скільки вона коштує в байтах та тактах.
*   **Створювати абстрактні класи** — проєктувати інтерфейси-контракти для гнучкої архітектури.
*   **Уникати витоків пам'яті** — знати, чому деструктор мусить бути віртуальним; керувати об'єктами на купі через **`std::unique_ptr`** (RAII).
*   Використовувати **`final`** — фіксувати контракт методу або заборонити подальше спадкування від класу.
*   Розуміти **RTTI** та **`dynamic_cast`** для безпечного звуження типу (downcast) від базового вказівника до похідного.

## Частина 1: Поліморфізм та Динамічне зв'язування

**Поліморфізм** (від грец. "багато форм") — це здатність одного і того ж виклику робити різні речі залежно від того, хто його виконує.

**Побутова аналогія:** Уявіть кнопку "Відтворити" (▶) на пульті. Ви натискаєте одну і ту ж кнопку, але:
*   Якщо пульт направлений на **телевізор** — починається фільм.
*   Якщо на **колонку** — грає музика.
*   Якщо на **проєктор** — світиться презентація.

Команда одна (`play()`), а поведінка різна — залежно від пристрою. В ООП "пристрій" — це тип об'єкта, а "кнопка" — виклик методу через вказівник на базовий клас. Але за замовчуванням C++ цього не вміє — давайте переконаємось.

### Крок 1: Створюємо проблему

Уявіть гру з різними типами юнітів. Ви створюєте базовий клас та нащадника:

```cpp
class Unit {
public:
    void move() { cout << "Unit: generic move" << endl; }
};

class Soldier : public Unit {
public:
    void move() { cout << "Soldier: tactical advance" << endl; }
};
```

Тепер ви хочете зберігати різних юнітів в одному масиві (через вказівник на базовий клас — це ж логічно, `Soldier` **є** `Unit`):

```cpp
Unit* squad[3];
squad[0] = new Soldier();
squad[1] = new Soldier();
squad[2] = new Unit();

for (int i = 0; i < 3; i++) {
    squad[i]->move();  // Що виведеться?
}
```

**Очікуємо:**
```text
Soldier: tactical advance
Soldier: tactical advance
Unit: generic move
```

**Отримуємо:**
```text
Unit: generic move
Unit: generic move
Unit: generic move
```

Всі три юніти рухаються як базовий `Unit`, навіть ті, що є `Soldier`. Солдати "забули", хто вони.

### Крок 2: Чому так? (Static Binding)

Компілятор дивиться на **тип вказівника** (`Unit*`), а не на **тип об'єкта** (`Soldier`). Він статично (на етапі компіляції) вирішує: "Тип — `Unit*`, отже викликаю `Unit::move()`". Це називається **Static Binding** — швидко, але "сліпо".

### Крок 3: Виправляємо — ключове слово `virtual`

Щоб компілятор визначав правильну версію методу **під час виконання** програми (а не під час компіляції), ми позначаємо метод батька як `virtual`:

```cpp
class Unit {
public:
    virtual void move() { cout << "Unit: generic move" << endl; }
};

class Soldier : public Unit {
public:
    void move() override { cout << "Soldier: tactical advance" << endl; }
};
```

> **Приклад в коді:** [oop.cpp](https://github.com/vplanto/cpp/blob/main/09-oop-intro/oop.cpp) (демонструє використання `virtual` та `override`, а також важливий нюанс — небезпеку рекурсії під час викликів).

Тепер той самий код з масивом виведе правильно:

```text
Soldier: tactical advance
Soldier: tactical advance
Unit: generic move
```

**Що змінилося:**
*   `virtual` у батька — каже компілятору: "Не вирішуй зараз, яку версію викликати. Переклади це рішення на рантайм."
*   `override` у нащадка — необов'язкове, але захищає від друкарських помилок (наприклад, `Move()` замість `move()` — компілятор одразу скаже про помилку).

### Downcasting: RTTI та `dynamic_cast`

**Upcasting** (від похідного до бази) у C++ звичайний: `Soldier*` без зусиль присвоюють у `Unit*`. **Downcasting** — навпаки: з `Unit*` дізнатися, чи це насправді `Soldier*`, і отримати вказівник на **конкретний** підтип. Це потрібно рідше, ніж віртуальні методи (часто достатньо викликати загальний `move()`), але трапляється, коли у `Soldier` є щось специфічне (боєприпаси, окремий UI), а контейнер тримає лише `Unit*`.

**RTTI** (*Run-Time Type Information*) — службова інформація про **фактичний** тип об'єкта під час виконання. Компілятор може її згенерувати для класів із поліморфізмом (найменше одна **`virtual`** функція в базі). На це спирається оператор **`dynamic_cast`**.

**Безпечний downcast вказівника** (на кшталт `Unit*` → `Soldier*`):

```cpp
Unit* p = squad[i];

if (Soldier* s = dynamic_cast<Soldier*>(p)) {
    // p дійсно вказував на Soldier (або його нащадка)
    // s->reload();  // приклад доступу лише до солдатського API
} else {
    // це не Soldier — коректно обробляємо без undefined behavior
}
```

Якщо `p` не вказує на об'єкт типу `Soldier` (і не на **публічного** нащадка `Soldier`), **`dynamic_cast<Soldier*>(p)` повертає `nullptr`**. Це відмінно від небезпечного **`static_cast<Soldier*>(p)`**, який «повірить» програмісту навіть там, де в пам'яті лежить звичайний `Unit` — і далі можливий крах при зверненні до полів солдата.

Для **посилань** форма `dynamic_cast<Soldier&>(*p)` кидає виняток **`std::bad_cast`**, якщо тип не збігається (бо посилання не може бути «порожнім»).

**Умови.** Базовий клас у ланцюзі `dynamic_cast` мусить бути **поліморфним** (хоча б одна віртуальна функція — у нашому прикладі це вже `virtual void move()`). Інакше компілятор відхилить `dynamic_cast` як невалідний для такого типу.

**Практика.** Не зловживайте downcast: якщо логіка роз’їжджається по `if (dynamic_cast<...>)`, часто варто додати **віртуальний метод** в `Unit` або застосувати **відвідувач / подвійну диспетчеризацію**. `dynamic_cast` залишається стандартним інструментом там, де звуження типу справді необхідне і рідкісне.

> У деяких збірках (наприклад, з прапором **`-fno-rtti`**) RTTI вимкнено — тоді `dynamic_cast` по таких типах недоступний. Для навчальних проєктів зазвичай RTTI увімкнений.

### `final` — кінець ланцюга перевизначень (C++11)

`override` каже: «я точно перевизначаю віртуальний метод батька». **`final`** відповідає на інше питання: «чи дозволено змінювати це *далі*».

**1. `final` на віртуальному методі** — заборона подальшого перевизначення в класах-нащадках. Корисно, коли реалізація в проміжному класі є остаточною (контракт, безпека, навмисне закриття гілки ієрархії).

```cpp
class Unit {
public:
    virtual void move() { /* ... */ }
};

class Soldier : public Unit {
public:
    void move() final override { /* рух солдата — далі не перевизначаємо */ }
};

// class EliteSoldier : public Soldier {
// public:
//     void move() override;  // помилка компіляції: move() у Soldier — final
// };
```

**2. `final` на самому класі** — заборона **будь-якого** спадкування від цього типу.

```cpp
class Utility final {
    // ...
};

// class Bad : public Utility { };  // помилка компіляції
```

**Коли це доречно:** «листкові» утиліти, типи з жорсткими інваріантами, свідоме закриття дизайну (немає підтипів — немає випадкових перевизначень і зайвих рівнів дерев класів). Це стандартна частина сучасного C++ поруч з `override`.

## Частина 2: Внутрішній механізм — V-Table

Як програма дізнається, яку функцію викликати, якщо ми працюємо через вказівник на базовий клас?

Коли в класі з'являється хоча б одна `virtual` функція, компілятор створює:
1. **V-Table** — приховану таблицю (масив вказівників на функції) для кожного класу.
2. **vptr** — прихований вказівник всередині кожного об'єкта, який вказує на відповідну V-Table.

**Ціна поліморфізму (Trade-offs):**
* **Пам'ять:** Кожен об'єкт стає більшим на 8 байт (через vptr) на 64-бітних системах.
* **Швидкість:** Замість прямого виклику функції, процесор робить "підвійний стрибок" (спочатку в vptr, потім за адресою з таблиці).

## Частина 3: Абстрактні класи (Інтерфейси) та Контракти

Іноді нам потрібен клас лише як "контракт" або шаблон поведінки. Наприклад, ми не можемо створити "просто фігуру" (`Shape`), ми можемо створити тільки конкретне Коло чи Квадрат.

```cpp
class Shape {
public:
    // Pure Virtual Function (= 0)
    virtual double getArea() const = 0; 
};
```

* **Pure Virtual Function (Чиста віртуальна функція):** Це метод із записом `= 0` в кінці. Він **не має реалізації** в базовому класі. Це просто обіцянка: "Усі мої нащадки точно вмітимуть це робити".
* **Абстрактний клас:** Клас, що містить хоча б одну чисту віртуальну функцію.
* **Заборона:** Ви не можете створити об'єкт цього класу (`new Shape()` — помилка компіляції), бо він ще "недописаний".
* **Обов'язок (Контракт):** Нащадки мусять реалізувати всі такі методи. Якщо `Circle` наслідується від `Shape` і не реалізує `getArea()`, `Circle` теж стає абстрактним класом і його не можна створити.

**Чому сучасне програмування йде у бік інтерфейсів?**
Спадкування реалізації (з готовим кодом) часто призводить до сильної зв'язності (High Coupling) — пам'ятаєте "Крихкий базовий клас"? Сучасні архітектури (як SOLID) рекомендують **наслідувати інтерфейси**, а не реалізацію.
Інтерфейс у C++ — це абстрактний клас, де *всі* методи є чистими віртуальними (`= 0`) і немає жодних полів даних. Це створює **слабку зв'язність** (Low Coupling): класи спілкуються лише через "контракти" поведінки, не знаючи, як воно працює всередині.

## Частина 4: Віртуальний деструктор (Важливо для пам'яті)

### Порядок конструкторів і деструкторів (з логами)

Для одного об'єкта нащадка діє **суворе** правило:

* **Конструктори** викликаються **зверху вниз** уздовж ланцюга спадкування: спочатку **базовий** клас, потім **нащадок** (і далі, якщо рівнів кілька).
* **Деструктори** — **дзеркально, знизу вгору**: спершу тіло **найглибшого** нащадка, потім його батько, і так до кореня.

Мінімальна програма з `cout`, щоб це **побачити** (можна скопіювати в `main.cpp` і запустити):

```cpp
#include <iostream>

class Base {
public:
    Base()  { std::cout << "1: Base ctor\n"; }
    ~Base() { std::cout << "4: ~Base dtor\n"; }
};

class Derived : public Base {
public:
    Derived()  { std::cout << "2: Derived ctor\n"; }
    ~Derived() { std::cout << "3: ~Derived dtor\n"; }
};

int main() {
    { Derived x; }
    return 0;
}
```

**Очікуваний вивід:**

```text
1: Base ctor
2: Derived ctor
3: ~Derived dtor
4: ~Base dtor
```

**Навіщо так зроблено.** Під час **конструювання** спочатку має існувати валідна «частина батька» — нащадок будується поверх уже ініціалізованої бази. Під час **знищення** спочатку розбирають те, що додав нащадок, поки базова частина ще жива; інакше `~Derived` не зміг би безпечно покладатися на поля/методи бази. Тому деструктори — у зворотному порядку до конструкторів.

**Чому конструктор не буває віртуальним.** Віртуальний виклик іде через **`vptr`**, а **`vptr` встановлюється під час виконання конструктора** (спочатку для типу бази, потім оновлюється для нащадка). Поки виконується конструктор бази, об'єкт ще **не є** повноцінним нащадком, тож «вибрати конструктор за поліморфним правилом» для типу, якого ще немає, мова не дозволяє — це не просто обмеження компілятора, а узгодженість із порядком будови об'єкта.

---

Це критичний момент при роботі з поліморфізмом. Якщо ви видаляєте об'єкт нащадка через вказівник на базу, ви можете отримати витік пам'яті.

```cpp
class Base { 
    virtual ~Base() { } // ПОВИНЕН бути віртуальним
};

class Derived : public Base {
    int* data = new int[100]; 
    ~Derived() { delete[] data; } 
};

// ...
Base* ptr = new Derived();
delete ptr; // Спочатку ~Derived, потім ~Base — але лише якщо ~Base віртуальний
```

Без `virtual` у `~Base` компілятор при `delete ptr` може згенерувати виклик **лише** `~Base()`, минувши `~Derived()` — порядок «знизу вгору» для нащадка **не виконається**, і пам'ять з `data` не звільниться.

> [!CAUTION]
> **Золоте правило:** Якщо у вашому класі є хоча б одна `virtual` функція — деструктор теж **мусить** бути віртуальним. Без цього `delete base_ptr` не викличе деструктор нащадка, і ресурси (пам'ять, файли) не звільняться.

### `Point` / `MovablePoint` на купі та `std::unique_ptr` (RAII)

Той самий сценарій, що й у [лекції 16](16_inheritance_polymorphism.md) на UML (`Point`, `MovablePoint` з `dx`, `dy`), але замість «ручного» `new` / `delete` — **володіння через `std::unique_ptr`**: при виході з області видимості викликається деструктор і пам'ять звільняється автоматично (принцип **RAII**). Для бази з віртуальними методами деструктор бази лишаємо **`virtual`**, щоб знищення через `unique_ptr<Point>` коректно пройшло через `~MovablePoint`, якщо колись зберігатимете нащадка під типом бази.

```cpp
#include <iostream>
#include <memory>

class Point {
protected:
    double x, y;
public:
    Point(double x = 0, double y = 0) : x(x), y(y) {}
    virtual ~Point() = default;

    virtual void print() const {
        std::cout << "Point(" << x << ", " << y << ")\n";
    }
};

class MovablePoint : public Point {
    double dx, dy;
public:
    MovablePoint(double x, double y, double dx, double dy)
        : Point(x, y), dx(dx), dy(dy) {}

    void print() const override {
        std::cout << "MovablePoint(" << x << ", " << y << ", "
                  << dx << ", " << dy << ")\n";
    }

    void move() {
        x += dx;
        y += dy;
        std::cout << "after move: (" << x << ", " << y << ")\n";
    }
};

void demo() {
    // було б: Point* p1 = new Point(1, 2); p1->print(); delete p1;
    std::unique_ptr<Point> p1 = std::make_unique<Point>(1, 2);
    p1->print();                           // виклик через operator->

    // було б: MovablePoint* mp1 = new MovablePoint(...); … delete mp1;
    std::unique_ptr<MovablePoint> mp1 =
        std::make_unique<MovablePoint>(1, 2, 3, 4);
    mp1->print();
    mp1->move();
}   // ~MovablePoint, ~Point викликаються автоматично — без delete
```

`std::make_unique` (C++14) створює об'єкт на купі й одразу загортає його в `unique_ptr`; **власник один**, копіювання заборонене, передача права — через `std::move`. Якщо ранній `return` або виняток переривають функцію, деструктор `unique_ptr` усе одно звільнить пам'ять — на відміну від коду з «забутим» `delete`.

## Частина 5: Множинне спадкування та Diamond Problem

C++ — одна з небагатьох мов, яка дозволяє наслідуватися від кількох класів одночасно.

```cpp
class Printer { void print(); };
class Scanner { void scan(); };
class MFP : public Printer, public Scanner { }; // Багатофункціональний пристрій
```

Але це породжує **Ромбоподібне спадкування (Diamond Problem)**:

```text
       Device (має поле isOn)
      /      \
 Printer    Scanner
      \      /
        MFP
```

Якщо `Printer` і `Scanner` наслідуються від `Device`, то `MFP` успадкує **дві копії** базового `Device`! У `MFP` буде два незалежних поля `isOn`. Якщо викличете `mfp.isOn`, компілятор видасть помилку: "Яку саме половину пристрою ви маєте на увазі?".

Для 1 курсу достатньо знати мінімальний синтаксис розв'язання через **віртуальне спадкування**:

```cpp
class Device { public: bool isOn = false; };

class Printer : virtual public Device { };
class Scanner : virtual public Device { };
class MFP : public Printer, public Scanner { };

int main() {
    MFP m;
    m.isOn = true;   // OK: у MFP одна спільна Device-частина
}
```

Суть `virtual public Device`: компілятор зберігає в кінцевому об'єкті (`MFP`) **одну** спільну копію `Device`, а не дві. За це платимо складнішою внутрішньою адресацією в пам'яті (додаткові службові вказівники/зсуви залежать від ABI), тому такий дизайн застосовують рідко й обережно.

> **Практичне правило для старту:** множинне спадкування стану тримайте як оглядову тему. У прикладному коді віддавайте перевагу композиції або множинному спадкуванню **інтерфейсів** (чистих абстрактних класів без полів).

## Частина 6: Підсумок зв'язків та Access Modifiers

### Таблиця: Спадкування, Композиція, Агрегація

Шпаргалка, щоб швидко обрати правильний зв'язок між класами:

| Тип зв'язку | Відношення | Хто керує життям? | Зв'язність (Coupling) | Приклад у коді |
|-------------|------------|------------------|------------------------|----------------|
| **Спадкування** | **Is-A** (Є чимось) | Батько і нащадок нероздільні | **Найсильніша** (Уникати, якщо можна) | `class Cat : public Animal` |
| **Композиція** | **Has-A** (Має / Частина) | Контейнер знищується → частина знищується | **Слабка** (Рекомендована) | `class Car { Engine e; };` |
| **Агрегація** | **Uses-A** (Використовує) | Частина живе незалежно від контейнера | **Найслабша** | `class Car { Driver* d; };` |

### Доступність при спадкуванні vs композиції (нагадування)

**Тип спадкування** (`: public`, `: private`, …) і **модифікатори членів** у базі (`public:`, `protected:`, `private:`) ми розбирали на початку [лекції 16](16_inheritance_polymorphism.md) — там таблиця та приклади, чому для Is-A пишуть саме `: public Base`.

Коротко для порівняння зв'язків:

*   **Спадкування:** у методах нащадка доступні **імена** `public` та `protected` членів батька; до **`private`** батька нащадок не звертається напряму (поля все одно лежать у об'єкті). Для **зовнішнього** коду видимість результату залежить від того, **`public`** це спадкування чи **`private`** — див. лекцію 16.
*   **Композиція / агрегація:** клас-"власник" викликає лише **`public`** API вбудованого або пов’язаного об'єкта — він **не** "внутрішній нащадок" і не має `protected` моста до полів партнера.

> **Золоте правило інкапсуляції:** не робіть поля даних `protected` або `public`. Поля — `private`; нащадку чи власнику давайте доступ через методи (гетери/сетери, обґрунтовані `protected` хуки).

## Бонус: Залишок принципів SOLID (S, I, D)

У попередній лекції ми розглянули Liskov Substitution (L) та Open/Closed (O). Тема абстрактних класів та поліморфізму дозволяє зрозуміти ще три фундаментальні ідеї:

**S — Single Responsibility Principle (Принцип єдиної відповідальності)**

Кожен клас (і кожен інтерфейс) повинен мати лише одну причину для зміни. Якщо ваш клас `Manager` одночасно рахує зарплату, підключається до бази даних і відправляє email — він порушує SRP і дуже скоро перетвориться на Spaghetti Code. Розділяйте логіку.

**I — Interface Segregation Principle (Принцип розділення інтерфейсу)**

Краще багато маленьких спеціалізованих інтерфейсів, ніж один "жирний" загальний.
*Помилка:* Створити `class IWorker { virtual void code()=0; virtual void sell()=0; };`. Програміст не вміє продавати, а менеджер не вміє кодити, але обидва змушені будуть малювати "пусті" методи (dummy overrides).
*Рішення:* Створити `iCoder` та `iSeller`. Класи реалізують лише те, що їм дійсно потрібно.

**D — Dependency Inversion Principle (Принцип інверсії залежностей)**

Верхні рівні архітектури не повинні залежати від нижніх. **Обидва мають залежати від абстракцій.**
*Помилка:* Клас `Game` (високий рівень) створює всередині себе об'єкт `OpenGLRenderer` (низький рівень). Щоб змінити графіку на DirectX, треба переписати `Game`.
*Рішення:* Створити абстрактний клас `IRenderer`. `Game` приймає вказівник на `IRenderer*` (через конструктор) і викликає `draw()`. Тепер `Game` не знає ні про OpenGL, ні про DirectX — він працює лише з *контрактом*. Це і є магія поліморфізму!

## Контрольні питання

1. Чому спадкування збільшує Coupling?
2. Що таке `override` і чому його варто писати, хоча це необов'язково?
3. Якщо `sizeof(int) = 4`, який розмір буде у класу:
```cpp
class A { 
    int x; 
    virtual void f(); 
};
```
(Підказка: згадайте про vptr і вирівнювання).
4. Чи може конструктор бути віртуальним? Який порядок виклику конструкторів і деструкторів у база→нащадок?

<details markdown="1">
<summary>Відповіді</summary>

1. Бо дочірній клас залежить від реалізації батьківського. Зміни в батьку можуть зламати нащадка.
2. `override` просить компілятор перевірити, чи існує такий віртуальний метод у батька. Це захищає від помилок у назвах (наприклад, `move()` vs `Move()`).
3. **16 байт.** `int` (4 байта) + `vptr` (8 байт на 64-bit). Через вирівнювання (padding) компілятор додає пустоту, щоб vptr був за адресою, кратною 8.
4. **Ні:** віртуальний виклик потребує `vptr`, а той встановлюється по ходу конструкторів — для «вибору» конструктора нащадка через таблицю віртуальних методів немає коректної моделі. **Порядок:** конструктори — **базовий → нащадок**; деструктори — **нащадок → базовий** (див. приклад з логами вище).

</details>
