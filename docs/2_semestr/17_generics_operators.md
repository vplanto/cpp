* **–ü–ª—é—Å:** –ù—ñ—è–∫–æ–≥–æ –æ–≤–µ—Ä—Ö–µ–¥—É –≤ Runtime (–Ω–∞ –≤—ñ–¥–º—ñ–Ω—É –≤—ñ–¥ Java/C#, –¥–µ —î boxing/unboxing). –®–≤–∏–¥–∫—ñ—Å—Ç—å —è–∫ —É –Ω–∞—Ç–∏–≤–Ω–æ–≥–æ –∫–æ–¥—É.

<details>
<summary>üî¨ <b>Mathematical View:</b> Templates as Type-Level Functions</summary>

**For Applied Mathematics Students:**

Let $\mathbb{T}$ denote the set of all valid C++ types:
$$\mathbb{T} = \{\text{int}, \text{float}, \text{double}, \text{UserDefinedTypes}, \dots\}$$

A `template <typename T> class Vector2D` is a **type-level function**:
$$F: \mathbb{T} \to \mathbb{T}$$

Usage `Vector2D<int>` is **function application**:
$$F(\text{int}) = \text{Vector2D}_{\text{int}}$$

**Monomorphization Explained:**

The compiler evaluates $F$ for each distinct argument used in the program.  
If you write:
```cpp
Vector2D<int> v_int(1, 2);
Vector2D<float> v_float(1.5, 2.5);
Vector2D<int> v_int2(3, 4);  // Reuses F(int)
```

The compiler generates:
$$\{F(\text{int}), F(\text{float})\} \subset \mathbb{T}$$

Only **2 distinct instantiations**, even though you created 3 objects.

**Why This Differs from Java Generics:**

- Java: `List<T>` ‚Üí One implementation with type erasure (runtime type checking)  
- C++: `Vector<T>` ‚Üí $n$ distinct implementations for $n$ distinct $T$ (compile-time generation)

**Trade-off:**
- ‚úì Zero runtime overhead (no boxing, no type checks)
- ‚úó Code bloat ($|F(\mathbb{T}_{\text{used}})|$ copies in binary)

**See also:** [Memory Model Glossary](00_memory_model_glossary.md#2-compilation--linking-model) for why templates must be in headers.

</details>
