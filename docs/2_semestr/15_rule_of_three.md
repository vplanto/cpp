# Лекція 15: Правило Трьох, Deep Copy та Перевантаження Операторів

[← Лекція 14](14_raii_lifecycle.md) | [Index](index.md) | [Далі: Лекція 16 →](16_inheritance_polymorphism.md)


## Мета
Зрозуміти найнебезпечнішу пастку C++ — **Shallow Copy** (поверхневе копіювання) і навчитися її уникати через "Святу Трійцю": Деструктор, Копі-Конструктор, Оператор Присвоєння.

## Експрес-опитування

1.  Ви дали другу ключі від своєї квартири. Чи означає це, що у нього з'явилася **власна** квартира, чи ви тепер просто ділите одну на двох?
2.  Ви орендуєте авто. У договору написано: "Повернути з повним баком". Що станеться, якщо ви просто кинете ключі в кущі й підете, не викликавши процедуру завершення оренди?
3.  Чому небезпечно мати два ідентичних пульти, які керують одним і тим самим телевізором, якщо кожен з них може його "вимкнути назавжди" (утилізувати)?

<details markdown="1">
<summary>Інженерна відповідь</summary>

1.  Це **поділ ресурсу (Shallow Copy)**. Якщо один змінить замок або продасть квартиру, інший залишиться ні з чим. В C++ це призводить до помилок, коли два об'єкти вважають, що володіють однією і тією ж пам'яттю.
2.  Це **витік ресурсу (Memory Leak)**. Процедура завершення оренди — це аналог **Деструктора**. Якщо її не викликати, ресурс (пам'ять/авто) залишиться заблокованим.
3.  Це проблема **Double Free**. Коли перший пульт "утилізує" телевізор, другий все ще думає, що об'єкт існує. Спроба другого пульта видалити вже неіснуючу річ призведе до фатальної помилки.

</details>

## Усі сценарії створення та копіювання об'єктів

Перш ніж розбирати проблеми, давайте систематизуємо, **коли і як** створюються чи копіюються об'єкти в C++.

| Синтаксис | Що відбувається | Який метод викликається | Примітка |
| :--- | :--- | :--- | :--- |
| `A a;` | Створення нового об'єкта | **Default Constructor** | Поля ініціалізуються |
| `A a(10);` | Створення з параметрами | **Parameterized Constructor** | - |
| `A a = b;` | Копіювання при створенні | **Copy Constructor** | Це створення нового об'єкта |
| `a = b;` | Перезапис існуючого | **Copy Assignment Operator** | Обидва об'єкти вже існують |
| `void f(A obj)` | Передача по значенню | **Copy Constructor** | Створюється копія на стеку |

### 💡 Чому ми наголошуємо на Константному Посиланні (`const &`)?
При передачі великих об'єктів (вектори, рядки) краще використовувати `const Type&`. Одне копіювання може тривати мілісекунди, а якщо їх тисячі — ваша програма "загальмує" на рівному місці. Також це захищає від нескінченної рекурсії: для створення копії за значенням Copy Constructor мав би викликати сам себе для створення копії параметра, і так до переповнення стека.

## Частина 1: Проблема Shallow Copy (Чому програма падає?)

Уявимо наш клас з динамічною пам'яттю. Він захоплює масив на початку і звільняє його в кінці:

```cpp
class Buffer {
    int* ptr;
public:
    Buffer(int value) { ptr = new int(value); }
    ~Buffer() { delete ptr; }
};

void crashTest() {
    Buffer b1(10);
    Buffer b2 = b1; // Тут компілятор робить "поверхневу" копію — просто копіює адресу
} // 1. b2 знищується, delete ptr — OK
  // 2. b1 знищується, знову delete той самий ptr — CRASH (Double Free)!
```

Це називається **Shallow Copy**. Вказівники скопіювалися, але дані — ні. Два власники на один ресурс завжди призводять до катастрофи, коли хтось перший його видалить.

## Частина 2: Copy Constructor (Клонування)

Щоб виправити це, ми маємо навчити C++ правильно клонувати наш об'єкт. Це називається **Deep Copy**.

**Синтаксис:** `ClassName(const ClassName& other)`

```cpp
class Buffer {
    int* ptr;
public:
    Buffer(int value) { ptr = new int(value); }
    ~Buffer() { delete ptr; }

    // Copy Constructor: Виділяємо НОВУ пам'ять
    Buffer(const Buffer& other) {
        // Виділяємо СВОЮ власну пам'ять і копіюємо вміст
        ptr = new int(*(other.ptr));
    }
};
```

## Частина 3: Operator Overloading (Вступ)

В C++ оператори (`+`, `-`, `=`, `<<`) — це просто функції зі смішними іменами. 
Щоб зрозуміти, як вони працюють, розділимо вираз `a + b` на дві частини:
* **LHS** (Left Side, `a`) — це об'єкт, **власник** методу. Він викликає оператор (прихований `this`).
* **RHS** (Right Side, `b`) — це значення, яке передається як **аргумент**.

Отже, запис `a + b` компілятор буквально перетворює на виклик методу: `a.operator+(b)`. Це дозволяє нашим класам поводитися природно, як вбудовані типи (`int`).

## Частина 4: Copy Assignment (Перезапис об'єкта)

При виразі `a = b` (коли обидва об'єкти вже існують), ми не просто копіюємо дані — ми маємо видалити старе, щоб звільнити місце для нового. Це найнебезпечніше місце.

**Чех-ліст безпечного присвоєння:**
1. **Self-check:** Чи не присвоюємо ми об'єкт самому собі (`a = a`)? Якщо так — нічого не робимо.
2. **Cleanup:** Видалити старі дані, які вже були в `LHS` (лівому об'єкті).
3. **Allocation:** Виділити нову пам'ять на купі.
4. **Copying:** Переписати дані з `RHS` (правого об'єкта) в нову пам'ять.
5. **Return this:** Повернути посилання на себе, щоб працювали ланцюжки (`a = b = c`).

```cpp
Buffer& operator=(const Buffer& other) {
    if (this == &other) return *this; // 1. Self-assignment check

    delete ptr;                       // 2. Cleanup
    ptr = new int(*(other.ptr));      // 3 & 4. Allocation + Copy
    return *this;                     // 5. Return this
}
```

## Частина 5: The Rule of Three (Свята Трійця)

**Золоте правило C++:** Якщо ви вручну керуєте пам'яттю (пишете Деструктор), ви **майже напевно** маєте реалізувати всі три методи:

1. **Destructor** — звільнити пам'ять.
2. **Copy Constructor** — щоб `new = old` не зламало програму.
3. **Copy Assignment** — щоб `exists = exists` не зробило double free.

Якщо ви пишете тільки деструктор, програма впаде на першому ж копіюванні.

## Частина 6: Move Constructor та Swap (Передача прав)

Іноді нам не потрібно копіювати дані (Deep Copy). Наприклад, якщо об'єкт-джерело є тимчасовим і скоро буде знищений. Набагато ефективніше просто **забрати** його ресурси. 

#### Move Constructor (Конструктор переміщення)
Це як передача естафети: ви не клонуєте атлета, ви просто забираєте у нього паличку.

```cpp
class Buffer {
    int* ptr;
public:
    // ... деструктор та інші методи
    
    // Переміщення: забираємо вказівник собі, а джерелу зануляємо його
    Buffer(Buffer&& other) noexcept : ptr(other.ptr) {
        other.ptr = nullptr; // Тепер інший об'єкт "порожній"
    }

    void swap(Buffer& other) noexcept {
        std::swap(ptr, other.ptr);
    }
};
```

#### Swap (Обмін)
Це найбезпечніший спосіб змінити дані. Замість того, щоб перемальовувати картину, два галеристи просто обмінюються рамами. Це працює миттєво, незалежно від розміру даних.

## Контрольні питання

1. Напишіть прототип Copy Constructor для класу `String`.
2. Чому в операторі присвоєння ми повертаємо `String&`, а не `void`?
3. Що станеться в `operator=`, якщо ми забудемо перевірку `if (this == &other)` і спробуємо скопіювати об'єкт у самого себе?
4. Які оператори **НЕ МОЖНА** перевантажити?

<details markdown="1">
<summary>Відповіді</summary>

1. `String(const String& other);`
2. Щоб підтримувати ланцюжок: `a = b = c;`. 
3. **Undefined Behavior / Crash.** Ми видалимо власні дані, а потім спробуємо прочитати їх же для копіювання.
4. Доступу до члена (`.`), Scope resolution (`::`), тернарний (`?:`) та `sizeof`.

</details>
