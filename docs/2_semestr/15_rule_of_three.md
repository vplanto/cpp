# Лекція 15: Правило Трьох, Deep Copy та Перевантаження Операторів

[← Лекція 14](14_raii_lifecycle.md) | [Index](index.md) | [Далі: Лекція 16 →](16_inheritance_polymorphism.md)


## Мета
Зрозуміти найнебезпечнішу пастку C++ — **Shallow Copy** (поверхневе копіювання) і навчитися її уникати через "Святу Трійцю": Деструктор, Копі-Конструктор, Оператор Присвоєння.

## Експрес-опитування
1. Якщо клас керує динамічною пам'яттю (RAII), що робить його деструктор?
2. Чи можна викликати деструктор вручну (наприклад, `obj.~MyClass()`)?
3. Що відбувається, коли ми пишемо `delete obj;` для динамічно створеного об'єкта (`MyClass* obj = new MyClass();`)?

<details markdown="1">
<summary>Інженерна відповідь</summary>

1. Деструктор звільняє захоплені ресурси (викликає `delete`, закриває файли тощо), запобігаючи витокам пам'яті.
2. Ніколи цього не робіть! Деструктор викликається компілятором автоматично при знищенні об'єкта (виході з області видимості). Ручний виклик призведе до повторного звільнення пам'яті (Double Free), коли компілятор викличе його вдруге "легально".
3. Спочатку викликається деструктор `~MyClass()` (щоб об'єкт міг коректно "попрощатися" з власними ресурсами), а вже потім звільняється пам'ять на купі (heap), яку безпосередньо обіймав сам об'єкт.

</details>

## Усі сценарії створення та копіювання об'єктів

Перш ніж розбирати проблеми, давайте систематизуємо, **коли і як** створюються чи копіюються об'єкти в C++.

| Синтаксис | Що відбувається | Який метод викликається | Примітка |
| :--- | :--- | :--- | :--- |
| `A a;` | Створення нового об'єкта "з нуля" | **Default Constructor** | Пам'ять виділяється, поля ініціалізуються |
| `A a(10);` | Створення з параметрами | **Parameterized Constructor** | - |
| `A a = b;` <br> `A a(b);` | Створення нового об'єкта як копії існуючого `b` | **Copy Constructor** | Це *завжди* створення нового об'єкта |
| `a = b;` | Перезапис **вже існуючого** об'єкта `a` даними з `b` | **Copy Assignment Operator** | Обидва об'єкти вже існують |
| `void f(A obj)` <br> `f(a);` | Передача копії об'єкта у функцію (Pass-by-value) | **Copy Constructor** | Створюється тимчасовий об'єкт `obj` на стеку функції |
| `A f()` <br> `return a;` | Повернення об'єкта з функції за значенням | **Copy Constructor** | Об'єкт копіюється з локальної області видимості назовні |

### 💡 Чому ми постійно наголошуємо на Константному Посиланні (`const &`)?
Згадайте Лекцію 7 ([Посилання](https://vplanto.github.io/cpp/07_references.html)). Ми говорили, що великі об'єкти треба передавати як `const Type&`. Тепер ви бачите **чому**:
1. **Продуктивність:** Кожен раз, коли ви передаєте об'єкт по значенню (`void process(A obj)`), C++ мовчки викликає **Copy Constructor**. Якщо це великий вектор або рядок — ви витрачаєте час і пам'ять на копіювання. Відповідно, передаючи `void process(const A& obj)`, ми уникаємо виклику Copy Constructor взагалі!
2. **Infinite Recursion в Copy Constructor:** Як ви побачите далі в цій лекції, Copy Constructor приймає аргумент саме за посиланням (`const A& other`). Чому так? Якби він приймав аргумент за значенням (`A other`), то для передачі цього значення під час створення копії йому довелося б викликати... знову Copy Constructor! Який би знову викликав Copy Constructor для створення копії, і так до нескінченності (Stack Overflow).

## Частина 1: Проблема Shallow Copy (Чому програма падає?)

Уявимо наш клас з динамічною пам'яттю (RAII з Лекції 14):

```cpp
class Buffer {
    int* ptr;
public:
    Buffer(int value) { ptr = new int(value); }
    ~Buffer() { delete ptr; }
};

void crashTest() {
    Buffer b1(10);
    Buffer b2 = b1; // Тут компілятор робить побітову копію!
} // Тут b2.dtor видаляє пам'ять. Потім b1.dtor видаляє ту ж пам'ять -> CRASH!

```

Це називається **Shallow Copy**. Вказівники скопіювалися, але дані — ні. Два власники на один ресурс.

---

## Частина 2: Copy Constructor (Клонування)

Щоб виправити це, ми маємо навчити C++ правильно клонувати наш об'єкт. Це називається **Deep Copy**.

**Синтаксис:** `ClassName(const ClassName& other)`

```cpp
class Buffer {
    int* ptr;
public:
    Buffer(int value) { ptr = new int(value); }
    ~Buffer() { delete ptr; }

    // Copy Constructor: Виділяємо НОВУ пам'ять
    Buffer(const Buffer& other) {
        std::cout << "Copying...\n";
        // 1. Беремо значення з іншого об'єкта
        int val = *(other.ptr);
        // 2. Виділяємо СВОЮ власну пам'ять
        ptr = new int(val);
    }
};

```

Тепер `Buffer b2 = b1;` створить повну незалежну копію.

## Частина 3: Operator Overloading (Вступ)

В C++ оператори (`+`, `-`, `=`, `<<`) — це просто функції зі смішними іменами. 
Щоб розібратися, як вони працюють під капотом, необхідно згадати концепцію **LHS** (Left-Hand Side) та **RHS** (Right-Hand Side), про яку ми говорили у [Лекції 6: Вказівники](06_pointers.md#lhs-та-rhs-ліва-та-права-сторони-операції).

У бінарному виразі, наприклад `a + b`:
* **LHS** (Ліва сторона, `a`) — це об'єкт, власник методу. Це об'єкт, **для якого** викликається оператор (він же прихований `this`).
* **RHS** (Права сторона, `b`) — це значення, яке передається у цей метод як **аргумент** (`other`).

Отже, запис `a + b` компілятор буквально перетворює на виклик методу на лівому об'єкті: `a.operator+(b)`. 

Завдяки цьому ми можемо навчити наші класи поводитися так само природно, як вбудовані типи (`int`, `float`).

```cpp
class Point {
    int x, y;
public:
    Point(int x, int y) : x(x), y(y) {}

    // Перевантаження +
    Point operator+(const Point& other) {
        return Point(this->x + other.x, this->y + other.y);
    }
    
    // Перевантаження ==
    bool operator==(const Point& other) {
        return this->x == other.x && this->y == other.y;
    }
};

Point p1(1, 2), p2(3, 4), p3(4, 6);
Point sum = p1 + p2; // Працює як sum = p1.operator+(p2);

// Ланцюжкові виклики (Chaining)
if (p1 + p2 == p3) { 
    // Як це бачить компілятор:
    // 1. Спочатку виконується ліва частина: temp = p1.operator+(p2)
    // 2. Потім результат порівнюється з правою: temp.operator==(p3)
    // Весь вираз цілком: (p1.operator+(p2)).operator==(p3)
}

```

### Які оператори НЕ МОЖНА перевантажити?
Хоча C++ дозволяє перевантажувати більшість операторів, деякі з них жорстко заблоковані на рівні мови, щоб уникнути хаосу в базовій логіці або парсингу синтаксису:
* `.` (Доступ до члена класу)
* `::` (Розширення області видимості / Scope resolution)
* `?:` (Тернарний умовний оператор)
* `sizeof` (Отримання розміру)
* `.*` (Доступ до члена через вказівник на член)

## Частина 4: Copy Assignment Operator (Найскладніший бос)

Тепер про `a = b`. Це не створення. Це **перезапис**.
Тут діє алгоритм:

1. Перевірити, чи не присвоюємо ми об'єкт самому собі (`if (this == &other)`).
2. **Звільнити** старий ресурс (ми ж його перезаписуємо!).
3. **Виділити** новий ресурс.
4. Скопіювати дані.
5. Повернути `*this` (щоб можна було писати `a = b = c`).

```cpp
// Усередині class Buffer
Buffer& operator=(const Buffer& other) {
    // 1. Self-assignment check
    if (this == &other) {
        return *this; // Нічого робити не треба
    }

    // 2. Звільняємо старе
    delete ptr;

    // 3. & 4. Виділяємо і копіюємо нове (Deep Copy)
    ptr = new int(*(other.ptr));

    // 5. Повертаємо посилання на себе
    return *this;
}

```

## Частина 5: The Rule of Three (Закон)

Якщо вашому класу потрібно вручну керувати ресурсом (писати Деструктор), ви **ЗОБОВ'ЯЗАНІ** реалізувати всі три методи:

1. **Destructor** (щоб звільнити ресурс).
2. **Copy Constructor** (щоб не було shallow copy при створенні).
3. **Copy Assignment Operator** (щоб не було shallow copy при присвоєнні).

Якщо ви напишете тільки деструктор — ваша програма впаде при копіюванні.

> **Modern C++ Note (Ідіома `NonCopyable`):** 
> Якщо ви відчуваєте, що об'єкт концептуально не повинен або не може бути скопійований (наприклад, мережеве з'єднання або унікальний файл), ви можете просто **заборонити** копіювання за допомогою ключового слова `= delete`.

```cpp
class NetworkConnection {
    int socketFd;
public:
    NetworkConnection() { socketFd = 123; /* З'єдналися */ }
    ~NetworkConnection() { /* Розірвали з'єднання */ }

    // Явно забороняємо компілятору генерувати методи копіювання:
    NetworkConnection(const NetworkConnection&) = delete; 
    NetworkConnection& operator=(const NetworkConnection&) = delete;
};

void testNode() {
    NetworkConnection a;
    // NetworkConnection b = a; // ❌ Помилка компіляції: use of deleted function
}
```

> Таким чином компілятор сам захистить вас від Shallow Copy, просто не дозволивши скомпілювати код, де відбувається несанкціоноване копіювання (саме так працює `std::unique_ptr`).

## Частина 6: Move Constructor та Swap (Коротко)

Іноді нам не потрібно копіювати дані (Deep Copy), оскільки об'єкт-джерело є тимчасовим (наприклад, повертається з функції) і скоро буде знищений. У таких випадках набагато ефективніше просто "вкрасти" його ресурси. Для цього використовується **Move Constructor** (Конструктор переміщення).

**Swap** (Обмін) — це спосіб обмінятися даними (вказівниками на пам'ять) між двома об'єктами без безпосереднього поглибленого копіювання. Функція swap є дуже корисною для реалізації безпечного від виключень оператора присвоєння (ідіома Copy-and-Swap).

```cpp
class Buffer {
    int* ptr;
public:
    Buffer(int value) { ptr = new int(value); }
    ~Buffer() { delete ptr; }

    // Конструктор переміщення (Move Constructor)
    // Приймає rvalue-посилання (&&)
    Buffer(Buffer&& other) noexcept : ptr(other.ptr) {
        // "Крадемо" ресурс і забираємо в іншого об'єкта посилання на нього
        other.ptr = nullptr; 
        std::cout << "Moved!\n";
    }

    // Власна функція swap
    void swap(Buffer& other) noexcept {
        // Міняємо місцями вказівники
        std::swap(ptr, other.ptr);
    }
};

void testMove() {
    Buffer b1(10);
    // std::move перетворює b1 на rvalue, вказуючи компілятору, що можна "вкрасти" ресурси
    Buffer b2 = std::move(b1); 
}
```
